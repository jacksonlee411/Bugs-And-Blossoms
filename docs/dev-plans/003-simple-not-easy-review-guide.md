# DEV-PLAN-003：AI 驱动开发的“简单而非容易”方案评审指引（Simple, Not Easy Review Guide）

**状态**: 规划中（2026-03-07 03:41 UTC）

> 本文档定位：当方案或代码由 AI 辅助生成时，用一套“按风险分级、按证据评审”的方法，把团队从“写得快（Easy）”拉回“结构清晰（Simple）”。本文适用于设计评审（dev-plan/Spec）与实现评审（PR），但**不替代** `AGENTS.md`、`Makefile`、CI 与各专题规范文档。

## 1. 背景与问题

AI 极大降低了“写出来”的门槛，但也会把复杂度从“显性设计”转移到“隐性耦合”：补丁式堆叠、复制粘贴、文件增殖、为过测试临时加分支、把历史包袱误当成标准模式继续扩散。

真正的风险不是“AI 写错一行”，而是“AI 让团队在不自觉中放弃结构化思考”，把原本可以简单表达的系统推向难以解释、难以替换、难以演化的状态。

## 2. 目标与非目标

### 2.1 核心目标
- [ ] 在评审中区分：方案是在追求“简单（Simple）”，还是只是在追求“容易（Easy）”。
- [ ] 把评审重点从“能不能跑”提升到“边界是否清晰、不变量是否明确、失败路径是否可解释、交付是否可验收”。
- [ ] 用统一的评审语言帮助 reviewer 快速发现纠缠、偶然复杂度与不必要抽象。
- [ ] 让设计阶段暴露结构性问题，而不是把关键决策拖到实现阶段靠试错解决。

### 2.2 非目标
- 不禁止使用 AI；本文关注的是“AI 产出如何被约束、解释与收敛”。
- 不要求所有改动都走同等重量的研究/计划流程；评审要求应与风险和影响面成比例。
- 不复制仓库级 stopline、命令矩阵和工具链细则；这些以 `AGENTS.md`、`Makefile`、CI 及对应专题 dev-plan 为单一事实源。
- 不要求“不同实现者必须写出同样结构的代码”；冻结的是边界、不变量、失败语义与验收结果，而不是实现层的每个细节。

## 3. 核心定义：Simple vs Easy

### 3.1 定义（工作共识）
- **简单（Simple）**：边界清晰、职责单一、依赖方向可预测；同一概念只有一种权威表达；关键状态和不变量可以被简明解释；变更具有局部性。
- **容易（Easy）**：以最小阻力把功能“塞进去”；通过堆叠分支、复制模式、文件增殖、兼容层和临时 fallback 回避设计决策；短期可运行，长期难理解、难维护、难演化。

### 3.2 评审开场三问
1. **边界与不变量**：这次改动触碰哪些边界？必须始终成立的规则是什么，由哪里承载？
2. **假设与选择**：关键假设是什么？若存在多种理解，是否已列出选项让用户选择？
3. **成功标准**：请求是否已转为可验证目标？是否存在更简单的可行方案？

## 4. 评审分级（按风险决定证据重量）

> 原则：不是所有改动都需要同样厚重的评审材料；越接近“契约、写路径、权限、租户、路由、生成物、AI 提交链路”，要求越重。

### 4.1 T0：轻量改动
适用场景：拼写、文案、纯格式、无外部行为变化的小重构。

最低要求：
- [ ] 能说明“不改变外部行为”。
- [ ] 不引入新边界、新契约、新依赖。
- [ ] 命中的最小门禁已执行。

### 4.2 T1：常规功能改动
适用场景：模块内功能增强、一般页面/接口改动、行为有变化但不触及核心不变量重塑。

最低要求：
- [ ] 有最小计划说明：目标、范围、边界、不变量、失败路径、验收方式。
- [ ] 明确命中哪些现有标准或门禁入口，并引用 SSOT。
- [ ] PR 能说明实现与计划的一致性。

### 4.3 T2：高风险改动
适用场景：API/数据库/迁移、权限/租户/RLS、路由治理、生成链路、One Door 写路径、No-Legacy 边界、AI 自动提交链路、多模型治理、外部依赖准入等。

最低要求：
- [ ] 在 `docs/dev-plans/` 以正式 dev-plan/Spec 冻结边界、契约、不变量、失败语义与验收标准。
- [ ] 明确演化路径、恢复语义、stopline 与门禁入口。
- [ ] 若涉及高风险提交链路或故障恢复路径，提供演练或等价验证证据；不把演练要求默认套用到所有改动。

## 5. 三阶段评审流程（Research → Plan → Implement）

> 目标：把“结构问题”尽量前移到 Plan 阶段解决；Implement 阶段主要做“对齐检查”。T0 可以跳过 Stage 1/Stage 2 的重型产物，T1/T2 按比例补齐。

### 5.1 Stage 1：研究（Research）——先认清现状，再决定是否需要改结构

**适用**
- T1：当改动跨多个边界，建议补简版现状说明。
- T2：必须有现状说明。

**最低输入**
- [ ] 现状概述：关键模块、包、路由、数据流、依赖方向。
- [ ] 约束清单：历史行为、权限、租户、时间语义、迁移、性能或生成物约束。
- [ ] 假设清单：当前方案依赖哪些事实、默认值、用户意图或现有不变量。
- [ ] 风险点：最容易出错的地方，以及如何验证。

**评审关注**
- 是否描述了真实现状，而不是“想象中的理想架构”。
- 是否分清了本质复杂度与历史包袱。
- 是否把关键困惑、歧义和待确认口径显式列出，而不是静默选择一种解释。

**停止线**
- [ ] 无法解释“现状为何如此”，却直接提出大改结构。
- [ ] 依赖与边界尚未弄清，就开始生成大量实现代码。
- [ ] 存在会影响契约、数据、权限、迁移或用户入口的歧义，却没有先澄清或先落 dev-plan。

### 5.2 Stage 2：计划（Plan）——冻结边界与验收，而不是冻结每个实现细节

**最低输入**
- [ ] 范围：Goals / Non-Goals。
- [ ] 假设：关键假设、被排除解释、未决问题和需要用户确认的点。
- [ ] 边界：模块职责、依赖方向、输入输出。
- [ ] 不变量：必须始终成立的业务或技术规则。
- [ ] 契约：接口、DTO、错误码、交互行为、数据约束。
- [ ] 失败语义：失败时如何拒绝、保护、恢复、重试或终止。
- [ ] 成功标准与验收标准：如何证明它是“简单且可交付”的，而不是只证明 tests pass；修 bug 需有复现路径，加校验需覆盖非法输入，重构需说明行为不变的验证入口。
- [ ] SSOT 对齐：引用命中的标准、门禁与专题规范，不在本文件或 dev-plan 中重复复制矩阵。

**评审关注**
- 计划是否足够确定：换一个实现者，也能得出**同边界、同不变量、同失败语义、同验收结果**。
- 复杂度是否被显式化：关键取舍是否在计划里说清，而不是留到实现阶段即兴决定。
- 是否主动比较了更简单的可行方案；若原请求会引入无必要复杂度，是否已推回并请用户确认。

**停止线**
- [ ] 计划只写“要做什么”，没写边界、不变量、失败路径、验收方式。
- [ ] 计划把关键一致性、并发、权限、恢复、租户语义藏在实现里，没有明确取舍依据。
- [ ] 需求存在多种合理解释，但计划没有说明解释差异、风险、推荐路径和需要用户选择的选项。

### 5.3 Stage 3：实现（Implement）——检查实现是否忠实履约

**最低输入**
- [ ] PR 指向对应 Spec/dev-plan，并声明偏离点。
- [ ] PR 说明关键假设与成功标准是否仍成立。
- [ ] 作者能解释主流程、失败路径、状态机和恢复语义。
- [ ] 本次命中的门禁与实际验证入口有最小证据。

**评审关注**
- 实现是否背离计划，在无文档更新的情况下临时发明了新边界或新抽象。
- 是否出现“为了过测”而引入的死分支、防御性分支或兼容层。
- 是否为已由上游不变量、类型/数据库约束、统一入口或门禁证明不可达的场景新增错误处理。
- 是否存在无关优化、相邻重构、格式噪音、没被要求的功能或未被请求的灵活性。
- 是否为只使用一次的代码新增抽象层、配置层、接口层或参数化能力。

**停止线**
- [ ] 新增了未在计划中出现的核心概念，却没有说明为何属于必要变化。
- [ ] reviewer 无法复述工作原理，而作者只能用“测试过了”作为主要论据。
- [ ] 需求、边界、数据语义或验收口径无法推导，却继续靠猜测实现。
- [ ] 存在更简单的可行方案但未说明取舍，或替用户决定了范围扩张。

## 6. 五维评审准则（Simple > Easy）

> 使用方式：每个维度至少给出“通过 / 警告 / 拒绝”之一，并附最小修改建议。只有明确触碰仓库级硬约束时，才应直接拒绝。

### 6.1 结构维度：解耦 vs 纠缠

**警告信号**
- [ ] 核心流程通过不断堆叠 `if/else` 吸收需求。
- [ ] 出现职责重叠的文件或类型增殖，只能靠“记忆”区分。
- [ ] 同一概念出现多套模型、DTO、错误码或校验路径，缺少权威来源。
- [ ] 为已证明不可达的场景新增 fallback、防御性分支或兼容错误路径。
- [ ] 为只使用一次的代码新增抽象层，或为了未来可能性提前引入接口/配置/参数化。

**通过标准**
- [ ] 边界清晰，职责能一句话定义。
- [ ] 依赖方向可预测，替换某模块不会引发大面积连锁改动。
- [ ] 抽象克制，不为“也许未来会用到”提前造层；只用一次的代码不建抽象层。
- [ ] 不可能场景的处理前移为不变量、约束、测试或断言，而不是散落在运行时分支中。

### 6.2 演化维度：规格驱动 vs 试错驱动

**警告信号**
- [ ] 方案主要来自多轮 Prompt 试错，最终实现混入大量防御性分支和兜底代码。
- [ ] 缺少验收标准，只能凭“看起来没问题”评审。
- [ ] 存在关键假设、用户意图或验收歧义，但实现中静默选择一种解释。
- [ ] 存在多种合理理解时，作者替用户做了范围或语义决定，而不是列出选项让用户选择。

**通过标准**
- [ ] 有与风险匹配的 Spec/dev-plan 或最小计划说明。
- [ ] 非平凡任务已在开工前转为可验证目标。
- [ ] 偏离点先更新文档，再更新实现。
- [ ] 测试、门禁和验收证据能对应到计划中的边界和不变量。

### 6.3 认知维度：本质逻辑 vs 偶然复杂度

**警告信号**
- [ ] 机械复刻历史模式，却说不清它保护了什么不变量。
- [ ] 复杂度的理由只是“仓库里一直这么写”。

**通过标准**
- [ ] 每段复杂逻辑都能回答：它保护了哪个不变量；若去掉，会破坏什么。
- [ ] 只服务于工具、历史兼容或旧接口的复杂度被隔离，并有退场条件。

### 6.4 维护维度：可理解 vs 仅可运行

**警告信号**
- [ ] 测试能过，但作者解释不清代码怎么工作。
- [ ] 修改一个点需要同时掌握过多隐式上下文。

**通过标准**
- [ ] 主流程、状态机、数据流和错误路径可被顺畅讲清。
- [ ] 命名与结构能形成连续叙事，而不是跳跃式拼图。

### 6.5 交付维度：可发现、可操作、可验收

**警告信号**
- [ ] 方案结构上很“优雅”，但用户无法在 UI 中发现或完成至少一条端到端操作。
- [ ] 所谓“后端先行”没有入口规划、占位、路由或明确验收方法。

**通过标准**
- [ ] 对用户可见的新能力满足 `AGENTS.md` 的“用户可见性原则”。
- [ ] 若短期只能后端先行，已明确用户入口规划与验收方式，避免形成僵尸功能。

### 6.6 过度设计维度：必要复杂度 vs 提前膨胀

> 过度设计不是“设计做多了”这么简单；它的核心问题是：把本应后移、可选或仅用于治理留痕的复杂度，提前写成首期范围、硬契约、硬验收或实现前置条件。它直接违反 `Simple > Easy`，因为它让系统更难解释、更难替换，也更容易把团队带回“先堆结构、再找价值”的路径。

**警告信号**
- [ ] 当前最小闭环尚未跑通，却先为未来场景补 capability、fallback、异步任务、第二套配置链或扩展点。
- [ ] 没人要求的“灵活性”“可配置”“可扩展”被写进首期范围。
- [ ] 为了“审计 / 治理 / 可观测”，把调试信息、过程证据或对象映射表上升为首期契约、数据面或验收前置。
- [ ] 将本可保持简洁的 prompt / summary / DTO 规格化为过重模板，导致实现、测试与演化成本显著上升。
- [ ] 先抽一整套接口、类型、层次和参数化配置，再回头寻找实际不变量和用户价值。
- [ ] 把“引用上游/复用上游”膨胀成对象级表格、证据矩阵、阶段 companion contract 和未填完不得实现的流程门槛。

**通过标准**
- [ ] 首期只冻结当前闭环真正需要的最小边界、不变量、失败语义与验收标准。
- [ ] 调试信息、治理留痕、远期扩展位、参数化能力明确分层：首期必需 / 可选增强 / P1 后移。
- [ ] 没被要求的功能不写；未被要求的灵活性和可配置默认不加。
- [ ] 新抽象、新字段、新任务、新配置项都能回答：若现在不做，会破坏哪个当前不变量；若答不出，应默认后移。
- [ ] 上游复用以“代码/模板/测试的实际承接关系”说明为主，不把流程性证据矩阵写成实现前置。

**常见过度设计类型 / 表现 / 示例**

| 类型 | 表现 | 示例 |
| --- | --- | --- |
| 流程治理型过度设计 | 把复用上游扩张成重流程、重证据、重门禁框架，实施前置条件比功能本身更重 | 要求每个上游对象补齐采用状态、原因类型、验证类型、PR/readiness 证据，并规定“未填完不得实现” |
| 提示词规格化过度设计 | 把本应简洁、可演进的 summary/prompt 写成固定栏目很多的业务交接模板 | compaction summary 被要求固定包含业务对象、日期、生效日、权限摘要、错误码、未决事项、禁止写入约束等 |
| 可观测性/审计扩张型过度设计 | 把本可后移的 telemetry/审计字段升级为首期架构、持久化和验收项 | 为首期 compaction 同时设计完整 telemetry 数据面、事件字段和硬验收要求 |
| 未来能力预留型过度设计 | 当前主链尚未封口，就先设计多 provider、多 runtime、远端能力和 fallback 规则 | 在只需本地 compaction 的阶段，先预留 `supports_remote_compaction` 与远端 fallback 策略 |
| 时序/任务编排型过度设计 | 在同步主链已足够时，再加异步任务、后处理任务或额外触发阶段 | 除 `manual compact` 和 `pre-turn auto compact` 外，再加入 `post-turn async precompact` |
| 抽象/参数化前置型过度设计 | 在问题域尚未稳定前，先抽很多接口、类型、配置项和参数化能力 | 一次性引入多层 `*Builder/*Planner/*Runner` 抽象，并把固定阈值提前做成模型/租户可配置 |

**评审动作**
- [ ] 对每一项新增复杂度标记其类别：当前不变量必需 / 治理留痕 / 未来扩展 / 纯属方便实现。
- [ ] 对“治理留痕 / 未来扩展 / 纯属方便实现”的复杂度默认提出后移建议，除非作者能说明其与当前不变量的直接关系。
- [ ] 若方案存在过度设计，但尚未触碰仓库级硬约束，通常按 Warning 收敛，而不是直接打回；重点是删减首期范围、降低前置条件、压缩抽象和验收口径。

## 7. Blocker 与 Warning 的边界

### 7.1 直接拒绝（Blocker）

以下情形应直接打回；具体门禁与执行入口以对应 SSOT 为准：

- [ ] 破坏仓库级硬约束：如第二写入口、legacy 双链路、租户或授权 fail-open、路由治理漂移、生成物漂移、密钥入库或入仓库。
- [ ] 命中 `AGENTS.md`、`DEV-PLAN-004M1`、`DEV-PLAN-005`、`DEV-PLAN-012`、`DEV-PLAN-017`、`DEV-PLAN-019`、`DEV-PLAN-021`、`DEV-PLAN-022` 等已冻结标准，却未对齐或无证据。
- [ ] 高风险改动没有对应计划文档，或计划缺失边界、不变量、失败语义、验收标准。
- [ ] 作者无法解释主流程与失败路径，只以“已经测试通过”作为主要依据。
- [ ] 涉及契约、数据、权限、迁移、用户入口或删除行为的关键困惑没有澄清，却继续按猜测实现。
- [ ] 存在多种会影响契约、数据、权限、迁移、用户入口或删除行为的合理理解，但没有列出选项让用户选择。

### 7.2 需要修改（Warning）

以下情形通常不应一票否决，但应要求收敛：

- [ ] 文件或类型命名让人困惑，但尚未形成第二套权威表达。
- [ ] 复用了旧模式，但仍能说明其不变量与适用边界。
- [ ] 有轻微样板代码或样式重复，但未显著破坏边界和局部性。
- [ ] 文档或 PR 证据不够清晰，但通过补充说明即可闭环。
- [ ] 为不影响契约的低风险歧义选择了实现路径，但未在 PR 或计划中写明假设。
- [ ] 为不可达场景添加了局部防御性分支；若不可达原因能前移为约束、测试或断言，应要求收敛。
- [ ] 增加了没被要求的功能、只用一次的抽象层，或没人要求的灵活性/可配置。
- [ ] 存在更简单的可行方案，但没有主动说明取舍或推回原方案。

## 8. 证据要求（按分级最小化）

### 8.1 T0 最小证据
- [ ] 说明为何不改变外部行为。
- [ ] 如存在假设，说明为何不会影响契约、数据、权限、迁移或用户入口。
- [ ] 给出命中的最小验证入口与结果。

### 8.2 T1 最小证据
- [ ] 说明本次假设、边界、不变量、失败路径与验收方式。
- [ ] 给出成功标准：修 bug 的复现路径、加校验的非法输入验证、重构的行为不变验证入口，或等价证据。
- [ ] 列出命中的标准或门禁入口，并引用 SSOT。
- [ ] 给出实际运行的验证入口与结果。

### 8.3 T2 最小证据
- [ ] 对应 dev-plan/Spec 链接。
- [ ] dev-plan/Spec 中已列出关键假设、多解释风险、推荐路径和需要用户确认的口径。
- [ ] 命中的标准与门禁入口说明。
- [ ] 实际执行的验证入口与结果。
- [ ] 若涉及生成物，提供生成入口与生成后一致性证据。
- [ ] 若涉及高风险恢复语义或 AI 提交链路，提供演练或等价验证证据。
- [ ] 若涉及对外契约变更，在 dev-plan 中先冻结边界与验收口径，并在 PR 中链接。

## 9. 快速“简单性”测试

1. **可替换性测试**：替换一个实现，不应引发跨层级、跨模块的连锁修改。
2. **局部性测试**：一个需求变化，应主要影响一个边界内的代码与测试。
3. **叙事测试**：从入口到结果的主流程可被连续描述，不需要反复跳层。
4. **不变量测试**：关键规则应由类型、约束、集中校验点或统一入口承载，而不是散落在许多条件分支里。
5. **假设测试**：任意关键假设都应能指出来源；若来源不明且会影响契约、数据、权限、迁移或用户入口，应先澄清。
6. **不可能场景测试**：若某错误路径声称是防御性兜底，应能说明它是否真的可达；不可达时应前移为约束、测试或断言。
7. **范围测试**：任意新增功能、抽象、配置项或灵活性都应能指向用户请求、当前不变量或既有边界；指不出则删除或后移。
8. **交付测试**：用户是否能发现、操作并验收这项能力；若不能，它是否至少有明确的入口规划与验证路径。
