# DEV-PLAN-226：测试指引 TG-004（门禁口径变更审批）

**状态**: 已完成（2026-03-03 01:09 UTC）

## 1. 背景

近期出现过通过调整覆盖率统计口径来满足门禁的情况。为避免“通过改口径替代补测试”，需要冻结测试门禁口径变更的审批边界。

## 2. 目标

1. [X] 建立测试指引 **TG-004**，明确门禁口径变更属于受控动作。  
2. [X] 明确“未经批准的门禁口径调整”为禁止事项。  
3. [X] 明确口径调整必须先获得项目负责人（你）的确认，再允许提交实现。

## 3. 适用范围（门禁口径定义）

以下内容均属于“测试门禁口径”，任何变更都受 TG-004 约束：

1. [X] 覆盖率策略文件：`config/coverage/policy.yaml`（阈值、统计范围、排除项）。  
2. [X] 覆盖率执行脚本：`scripts/ci/coverage.sh`（`coverpkg` 计算、测试包选择逻辑）。  
3. [X] CI 执行条件：`.github/workflows/quality-gates.yml` 中 `make test` 的触发条件。  
4. [X] 路径触发器：`scripts/ci/paths-filter.sh` 中与测试执行相关的匹配规则。

## 4. 强制规则（TG-004）

1. [X] **禁止事项**：未经批准，不得新增/扩大任何覆盖率排除项，不得降低阈值，不得缩小测试触发范围。  
2. [X] **审批前置**：凡涉及门禁口径变更，必须先获得你（项目负责人）的明确确认。  
3. [X] **证据要求**：PR 描述必须包含“变更前后口径差异、影响范围、实测结果、审批记录链接/引用”。  
4. [X] **实施顺序**：先确认、后改动；未确认状态下只允许补测或修复代码，不允许先改门禁口径。  
5. [X] **冲突处理**：若临时修复与 TG-004 冲突，以“保持既有门禁口径 + 补测”优先。
6. [X] **套件同步**：当新计划完成实施或既有功能完成调整时，必须同步调整或新增 `DEV-PLAN-060` 对应的全链路业务测试套件，确保主测试套件与最新业务契约一致。
7. [X] **死分支处理优先级**：当 100% 覆盖率门禁遇到“疑似死分支”时，优先执行“证明后删除”；仅对真实可达分支补测试，不得以调整覆盖率口径、扩大排除项或伪造 fallback 代替。
8. [X] **保留分支的条件**：若分支承担明确业务兜底语义且不可直接删除，必须通过职责拆分、纯函数化、接口注入等方式提升可测性，并在对应计划/记录中说明原因。
9. [X] **环境口径不得漂移**：当仓库已冻结外部依赖运行形态（例如 PostgreSQL 运行于 Docker / compose）时，测试实现不得新增“宿主机额外工具必装”前置条件；如需直连数据库，必须优先走容器内工具链，而不是将宿主机缺少工具误判为业务失败。
