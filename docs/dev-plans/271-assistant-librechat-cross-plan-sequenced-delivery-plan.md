# DEV-PLAN-271：Assistant/LibreChat 跨计划分阶段推进与封板编排计划（223/240/260/280）

**状态**: 进行中（2026-03-08 CST；`223/235/281/282/283/240A/240B/284/286` 已完成，当前执行焦点收敛为 `287 -> 289`，`285` 仍保持封板尾项）

## 1. 背景与立项原因
1. [ ] `DEV-PLAN-240` 已完成 `M0/M1` 契约冻结，但 `M2+` 进入实现阶段前，仍需要与 `223/260/280/284` 建立统一排程与 stopline。
2. [ ] `DEV-PLAN-260` 已冻结 P0 DTO/FSM 契约，但 Case 1~4 的真实业务闭环仍依赖 `240` 编排落地、`266/284` 承载面收口与 `223` 事实源稳定输出。
3. [ ] `DEV-PLAN-280` 的 `281/282/283` 已完成，`284` 进入正式 patch 前需要与 `240/260/223` 的入口条件完全对齐，避免前后端并行时返工。
4. [ ] `237` 与 `285` 属于封板尾项，不应提前吞并实现缺口；若无统一跨计划路线图，执行上容易退化为“边实现边封板”。

## 2. 目标与非目标

### 2.1 核心目标
1. [ ] 形成单一执行主链，并把父计划编号映射到可直接派工的执行单元。
2. [ ] 冻结阶段入口/出口，避免 `240/260/266/284/285/237` 之间出现“状态已写完成，但前置未满足”的漂移。
3. [ ] 明确默认顺序与可选并行条件；默认以稳定优先，只有满足条件时才允许局部并行。
4. [ ] 明确封板边界：`285` 只做复核与封板，不承担剩余实现。

### 2.2 非目标
1. [ ] 本计划不替代 `223/240/260/266/280/284/285/237` 的业务契约正文。
2. [ ] 本计划不新增第二事实源、第二写入口、旧入口兜底或任何 legacy 双链路。
3. [ ] 本计划不重复定义 schema 字段、UI patch 细节、MCP 提示词或测试脚本细节；这些仍由对应子计划承接。
