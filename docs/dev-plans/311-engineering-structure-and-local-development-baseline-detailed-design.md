# DEV-PLAN-311：工程结构与本地开发基线详细设计

**状态**: 规划中（2026-03-18 15:34 CST）

## 1. 背景与定位

本计划是 [DEV-PLAN-310](/home/lee/Projects/Bugs-And-Blossoms/docs/dev-plans/310-engineering-quality-testing-and-delivery-plan.md) 的 `311` 子计划，同时承接：

- [DEV-PLAN-300](/home/lee/Projects/Bugs-And-Blossoms/docs/dev-plans/300-greenfield-csharp-hr-platform-functional-blueprint.md) 对“模块化单体 + C#/.NET + React + PostgreSQL + Hangfire + Linux 容器平台”的冻结；
- [DEV-PLAN-400](/home/lee/Projects/Bugs-And-Blossoms/docs/dev-plans/400-implementation-roadmap-and-vertical-slice-plan.md) 对“以垂直切片而非模块堆叠作为主要交付节奏”的冻结；
- 后续 `340/350/360/370/380/390` 对统一工程结构、本地运行方式、最小复现链路与种子数据入口的共同依赖。

`310` 已经说明：如果没有单独的工程基线，后续所有子计划都会把“目录怎么放、模块怎么起、开发环境怎么拉起来、样例数据怎么灌、前后端怎么联调”留到实现时各自决定，最后形成：

- 目录层次漂移；
- 本地依赖启动方式分叉；
- 同一个问题在不同模块无法按同一路径复现；
- 种子数据、演示数据与测试夹具彼此混写；
- 平台壳、业务模块、后台任务各自维护第二套运行入口。

`311` 的职责就是把这些问题收敛为 **Greenfield 工程结构与本地开发基线 SSOT**，为 `312/313` 和所有业务子计划提供统一起跑线。

## 2. 目标与非目标

### 2.1 核心目标

- [ ] 用“业务规则优先”的语言重述工程结构与本地开发，不让 `.sln`、目录碎片、脚本命名或 IDE 偏好成为主叙事。
- [ ] 冻结 solution / repo / module 的 ownership 结构，明确哪些目录承载平台、业务域、共享能力与运行入口。
- [ ] 冻结本地开发的最小运行闭环：API、Web、Worker、数据库、文件存储、种子数据与本地配置引用的基本关系。
- [ ] 冻结本地 bootstrap、环境切换、样例数据/演示数据/测试夹具的边界，避免后续子计划继续混写。
- [ ] 为 `312` 提供统一的测试夹具与本地复现输入，为 `313` 提供统一的构建与部署物输入。
- [ ] 为 `340/350/360/370/380/390` 提供统一工程落点，避免各模块自造第二套运行入口或目录规则。

### 2.2 非目标

- [ ] 本计划不定义测试金字塔、覆盖率口径、E2E 编排细节；这些由后续 `312` 承接。
- [ ] 本计划不定义 CI/CD、`OCI image`、Linux 容器发布流水线与观测基线细节；这些由后续 `313` 承接。
- [ ] 本计划不直接定义业务 schema、API 契约或页面规则。
- [ ] 本计划不要求第一阶段就把本地环境做成“缩小版生产集群”，也不引入超出 `300` 蓝图的重型基础设施。
- [ ] 本计划不允许为了本地方便而引入默认租户、共享密钥、假权限旁路或第二套运行事实源。

## 3. “业务规则优先”在工程结构与本地开发中的翻译

### 3.1 开发者真正关心的是“如何稳定开工与复现”，不是某个目录名本身

开发者真正需要回答的是：

- 新人拉下仓库后，如何在最短路径内跑起第一条切片；
- 某个问题发生时，应该到哪个入口复现，而不是在多个脚本之间猜；
- 某个模块应该把代码放在哪里、由谁拥有，而不是凭感觉扩目录；
- 哪些数据是样例数据，哪些是测试夹具，哪些是演示环境种子；
- 前端、后端、后台任务是否围绕同一套运行上下文工作。

### 3.2 工程结构首先回答 ownership，不是美观

`311` 冻结：

- 目录结构首先表达 bounded context、共享层与运行入口；
- 运行入口首先表达产品边界，而不是工具偏好；
- 一个目录必须回答“谁拥有、面向谁、是否允许被别的模块直接依赖”。

### 3.3 本地开发首先回答“能否重现产品边界”，不是“能否把一切塞进单进程”

本地环境需要优先保证：

- 能看见租户、身份、时间视图、后台任务与回执这些系统边界；
- 能用最小依赖跑通垂直切片；
- 能在不引入额外漂移的前提下做联调、复现与 smoke。

### 3.4 本地方便不得改写平台合同

以下便利路径默认禁止：

- 本地直连跳过租户边界；
- 在前端或脚本里私存第二套“当前用户/当前租户”真值；
- 用演示假数据掩盖真实时间/权限/回执语义；
- 通过提交本地 secret 或共享测试账号来换取启动便利。

## 4. 当前基线：已沉淀的共享结论

### 4.1 已稳定的 Greenfield 方向

- [DEV-PLAN-300](/home/lee/Projects/Bugs-And-Blossoms/docs/dev-plans/300-greenfield-csharp-hr-platform-functional-blueprint.md) 已明确：
  - 默认形态是模块化单体；
  - 标准发布物是 `OCI image`；
  - 默认部署基座是 Linux 容器平台；
  - 平台、业务域、数据工作台与 Assistant 都是正式一等模块。
- [DEV-PLAN-310](/home/lee/Projects/Bugs-And-Blossoms/docs/dev-plans/310-engineering-quality-testing-and-delivery-plan.md) 已明确：
  - 本地开发、测试分层、CI/CD、容器发布与观测属于同一条工程主线；
  - `311/312/313` 应分别拥有工程结构、测试策略与交付观测三类问题域。
- [DEV-PLAN-400](/home/lee/Projects/Bugs-And-Blossoms/docs/dev-plans/400-implementation-roadmap-and-vertical-slice-plan.md) 已明确：
  - 交付节奏以真实垂直切片为主；
  - 工程基线应服务于“尽快跑通第一条链路”，而不是先堆一整套抽象。

### 4.2 当前主要缺口

1. [ ] **缺少 solution / repo ownership 的单一事实源**  
   现在只有 `300` 的模块结构建议，但还没有冻结“代码目录、运行入口、共享层、测试目录、开发支撑目录”之间的 ownership。

2. [ ] **缺少本地运行形态的共享合同**  
   目前只知道未来交付物是 Linux 容器，但没有单独定义本地 API / Web / Worker / DB / 文件存储应如何组成最小闭环。

3. [ ] **缺少种子数据、演示数据与测试夹具的分层边界**  
   若不提前冻结，后续很容易把 demo data、manual seed 与 test fixture 混写成一团。

4. [ ] **缺少新人开工与问题复现的标准路径**  
   没有统一的 bootstrap 与 reset 语言，后续 bug 排查会高度依赖口头传承。

## 5. 工程结构与本地开发基线蓝图

### 5.1 领域使命

`311` 是 Greenfield 平台内“**代码应该落在哪里、本地运行应如何组成、开发者如何稳定开工、如何以统一路径复现与联调**”的共享工程权威。  
它不拥有测试口径，也不拥有发布流水线，但为二者提供共同地基。

### 5.2 核心工程对象

| 工程对象 | 工程含义 | 是否由 `311` 拥有共享合同 |
| --- | --- | --- |
| `SolutionTopology` | 仓库、solution、模块、共享层与运行入口的结构蓝图 | 是 |
| `ModuleWorkspace` | 单个模块的标准分层、依赖方向与落点约定 | 是 |
| `RuntimeEntry` | `Web / Api / Worker` 等本地与交付入口的运行职责 | 是 |
| `LocalRuntimeProfile` | 本地最小运行闭环的依赖组合与配置画像 | 是 |
| `DeveloperBootstrap` | 新人开工、环境启动、数据初始化与 reset 的标准路径 | 是 |
| `SeedDataset` | 演示样例、联调样例与最小平台基线数据 | 是 |
| `TestFixtureDataset` | 供 `312` 与自动化使用的测试夹具合同 | 否，`312` 拥有测试语义，`311` 只定义边界 |
| `SecretReference` | 本地配置如何引用 secret，而不是直接提交 secret 值 | 是（共享边界） |

### 5.3 面向开发者的主能力

- 在统一目录结构中快速定位平台层、业务层、共享层与运行入口；
- 按统一 bootstrap 路径拉起 API、前端、后台任务与最小依赖；
- 加载最小平台种子数据，进入第一条垂直切片；
- 以统一方式复现租户、时间、权限、回执与异步任务问题；
- 在不引入目录漂移的前提下添加新模块、新页面或新任务；
- 为 `312/313` 提供稳定的本地执行底座。

## 6. `311` 冻结的目标规则矩阵

| 场景 | 开发者真正要做什么 | 核心工程规则 | 工程结果 |
| --- | --- | --- | --- |
| 新人开工 | 回答“仓库拉下后怎么尽快跑起来” | 必须存在统一 bootstrap 与最小样例数据路径；不得依赖口头步骤 | 第一条切片可复现 |
| 新增模块 | 回答“代码放哪、入口在哪、谁拥有” | 新目录必须落在既定模块层次内；不得临时长出第二套应用入口 | ownership 清晰 |
| 本地联调 | 回答“前后端与 worker 如何围绕同一上下文运行” | API、Web、Worker 必须围绕同一租户/会话/回执事实源联调 | 联调路径统一 |
| 复现问题 | 回答“这个 bug 如何在本地稳定重现” | 问题复现必须能绑定明确环境画像、种子数据与运行入口 | 排障路径可复用 |
| 种子数据初始化 | 回答“哪些数据用于平台最小闭环” | 样例种子、演示数据、测试夹具必须分层，不能混作一套 | 数据语义清楚 |
| 本地配置 | 回答“哪些值可提交、哪些只能本机引用” | secret 只允许本地引用，不允许进仓；缺失高风险配置时必须 fail-closed | 配置边界清晰 |

## 7. 共享合同、不变量与实现护栏

### 7.1 Solution Topology 合同

`311` 建议至少冻结以下顶层 ownership：

```text
src/
  Platform/
  HR/
  Coordination/
  DataWorkbench/
  Assistant/
  Shared/
apps/
  Web/
  Api/
  Worker/
tests/
  Unit/
  Integration/
  E2E/
dev/
  seeds/
  fixtures/
  scripts/
```

冻结原则：

- 平台能力、业务域能力、横切共享层和运行入口必须分层表达；
- 不允许业务模块把平台壳、后台任务或共享能力直接塞回模块私有目录；
- `Shared` 只承载共享合同、helper 与 cross-cutting abstraction，不承载第二套业务所有权。

### 7.2 Module Workspace 合同

- 单模块内部默认遵循 `Domain / Application / Infrastructure / Web` 分层。
- 新模块必须先回答 ownership，再申请目录落点。
- 同一业务能力不得同时在 `Platform/*` 与 `HR/*` 各拥有一套主实现。

### 7.3 Local Runtime Profile 合同

- 本地最小闭环至少要能表达：`Web + Api + Worker + PostgreSQL + 文件存储等价物`。
- 本地运行配置必须支持明确区分：
  - API 入口；
  - 前端入口；
  - 后台任务入口；
  - 种子/初始化入口。
- 本地最小闭环优先服务第一条垂直切片，而不是追求把全部生产依赖完整镜像。

### 7.4 Seed 与 Fixture 分层合同

- `SeedDataset` 回答“为了进入平台最小闭环，需要哪些租户、用户、字典与演示对象”。
- `TestFixtureDataset` 回答“自动化测试如何构造确定性输入”，由 `312` 拥有测试语义。
- 演示数据不得偷偷承担测试断言职责；测试夹具也不得被当作人工操作样板。

### 7.5 配置与 Secret 合同

- 仓库只保存可公开提交的默认配置与 `SecretReference` 约定。
- 本地高风险配置只允许存在于本机私有入口，不允许提交到仓库。
- 缺失租户、模型 provider、数据库凭据等关键配置时必须 fail-closed。

### 7.6 本地复现与调试合同

- 本地问题复现应能绑定明确的：
  - 租户；
  - 主体；
  - 时间视图；
  - 请求/回执标识；
  - 样例数据版本。
- 后台任务、导入导出、Assistant 动作等长事务问题，本地也必须有稳定查询入口，而不是只靠控制台日志猜测。

### 7.7 stopline

- 不允许提交本地 secret、共享测试账号或私钥材料。
- 不允许模块自行引入第二套启动入口、第二套 worker 框架或第二套本地事实源。
- 不允许把 demo data、测试夹具、开发临时数据混写为一套“万能样例”。
- 不允许通过本地默认租户、默认超级权限或跳过确认来伪造真实产品边界。

## 8. 作为后续子计划的业务需求输入

### 8.1 对 `312`（测试金字塔与 E2E 策略）的输入

- [ ] `312` 应直接消费 `311` 冻结的运行入口、种子数据边界与本地复现路径。
- [ ] 测试夹具可以扩展 `311` 的 `SeedDataset`，但不得回写成第二套 bootstrap 主链。

### 8.2 对 `313`（CI/CD、部署与观测基线）的输入

- [ ] `313` 的构建单元、容器入口、smoke 目标与运行画像必须建立在 `311` 的 solution topology 与 runtime entry 之上。
- [ ] 本地运行入口与交付运行入口应保持同一 ownership 语言，不能各自重新命名。

### 8.3 对 `340/350` 的输入

- [ ] `340` 的平台基座与 `350` 的前端壳层应使用 `311` 定义的 `Api / Web / Worker` 入口边界，不得各自再发明启动面。
- [ ] 租户、用户、平台种子数据需要有统一的本地初始化路径，支撑登录与产品壳最小闭环。

### 8.4 对 `360/370/380/390` 的输入

- [ ] 所有业务模块都必须落在统一模块 ownership 与 shared contract 结构中。
- [ ] 导入导出、后台任务、Assistant 编排与回执观察在本地应共享同一条复现与联调基线，而不是各自维护私有脚本。
