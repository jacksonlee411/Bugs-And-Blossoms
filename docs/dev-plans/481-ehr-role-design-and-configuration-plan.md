# DEV-PLAN-481：EHR 角色定义与极简用户授权方案

**状态**: 规划中（2026-04-28 18:55 CST）

## 0. 适用范围与评审分级

- **评审分级**：`T2`
- **范围一句话**：冻结角色定义与用户授权的最小交互边界；角色只定义功能权限，用户授权页顶部选择用户，主区用“角色 / 组织范围”两个页签维护授权表格。
- **关联模块/目录**：`pkg/authz/**`、`config/access/**`、`scripts/authz/**`、`internal/server/**`、`apps/web/src/**`
- **关联计划/标准**：`AGENTS.md`、`DEV-PLAN-000`、`DEV-PLAN-001`、`DEV-PLAN-012`、`DEV-PLAN-022`、`DEV-PLAN-032`、`DEV-PLAN-480`
- **用户入口/触点**：角色列表、角色详情、角色定义表单、用户角色分配、组织范围选择器、授权摘要查询

### 0.1 Simple > Easy 三问

1. **边界**：角色定义 owner 是基础信息和功能权限；用户授权 owner 是用户选择、角色行和组织范围行。字段安全、有效期、冲突检测、策略预览不进入本轮极简授权界面。
2. **不变量**：角色定义页不出现组织范围，不要求管理员为每个 action 手工维护 `scope_required`；组织范围由资源/能力 registry 的系统元数据推导，并在用户授权页提示和校验。
3. **可解释**：管理员先定义角色能做什么；进入用户授权页后先选择用户，再在角色页签添加角色行，在组织范围页签添加组织范围行。保存即生效，不再设置“保存并发布”或发布阻断流程。

### 0.2 现状研究摘要

- `DEV-PLAN-480` 已冻结 EHR 授权体系蓝图，但早期草图把“范围要求”“字段策略”“完整性校验”放在角色设计页，容易把功能权限、数据范围和字段级安全强耦合。
- 当前 `config/access/policies/**` 与 Casbin 工具链只覆盖 capability/API 级授权，不覆盖在线角色定义、范围绑定或保存生效工作流。
- 更符合管理员心智的分层是：角色定义回答“能做什么”；用户授权回答“谁在什么组织范围内行使这个角色”。
- 本方案先冻结交互与契约，不直接新增 DB 表、迁移或在线写入实现；任何持久化 SoT 后续必须另起实施计划并获得 schema 确认。

## 1. 背景与上下文

EHR 里的“角色”不是“给某个用户的一次授权记录”。角色应该表达一类职责的稳定功能权限模型，例如“鲜花公司 HR”“集团 HR”“组织只读用户”。

一个角色只回答：

- 角色名称、标识与描述是什么。
- 这个角色具备哪些功能权限，例如组织查看、组织导出、员工查看、薪酬报表查看。
- 这个角色是否可保存，是否存在无效 capability key、重复 key 或基础信息缺失。

以下内容不属于角色定义：

- “用户 B 能看鲜花事业部及下级”：这是用户授权/角色分配时绑定的数据范围。
- “组织资源需要组织范围”：这是资源/能力 registry 的系统元数据，不应让角色设计者逐项手工声明。
- “字段隐藏、脱敏、有效期、冲突检测、策略预览”：这些不是本轮极简授权界面的交付项。

## 2. 目标与非目标

### 2.1 核心目标

1. [ ] 冻结职责边界：角色定义只包含基础信息和功能权限；用户授权只包含主体、角色引用和组织范围绑定。
2. [ ] 定义角色定义 UI：角色基本信息、资源-操作矩阵、单一保存按钮，保存即生效。
3. [ ] 定义用户授权 UI：页面顶部选择用户；主区拆成“角色”和“组织范围”两个页签；每个页签都是可加行表格。
4. [ ] 明确校验边界：角色保存只校验角色自身合法性；授权保存只校验必填组织范围。

### 2.2 非目标

1. 不在本计划中新增 DB schema、迁移、在线角色写入 API 或后台生效工作流。
2. 不让角色定义页维护组织范围或 `scope_required`。
3. 不把字段策略硬编码在角色定义里，也不在用户授权页选择、编辑或预览字段安全配置。
4. 不在极简授权界面提供有效期、冲突检测、策略预览、多步骤流程或多维度并行设置。
5. 不替换 Casbin，不引入 OPA/CEL 或第二套 policy engine。
6. 不恢复 legacy、SetID、scope/package 或旧 org_level/scope_type/scope_key 语义。

### 2.3 用户可见性交付

- **用户可见入口**：
  - `系统管理 > 角色管理 > 新建/编辑角色`：只编辑角色基础信息和功能权限。
  - `系统管理 > 用户管理 > 用户授权`：选择用户后，在“角色”和“组织范围”两个页签中分别维护授权行。
  - 授权摘要查询：查看某个用户最终具备哪些能力和哪些数据范围。
- **最小可操作闭环**：
  - 管理员创建 `flower-hr`，勾选 `orgunit.orgunits:read`、`orgunit.orgunits:export`、`employee.employees:read`、`salary.reports:read`。
  - 管理员保存角色时只校验名称、slug、capability key 等角色自身完整性；保存即生效。
  - 管理员在用户授权页顶部选择用户 B。
  - 管理员在“角色”页签添加 `flower-hr` 行，角色说明列展示该角色可查看或操作的内容。
  - 管理员在“组织范围”页签添加“鲜花事业部”行，“包含下级组织”列初始为已选中。

## 3. 概念模型与边界

### 3.1 核心概念

| 概念 | 含义 | 归属 |
| --- | --- | --- |
| Role Definition | 角色定义：名称、slug、说明、能力集合、修订号 | 481 |
| Capability | 稳定 `object:action`，例如 `orgunit.orgunits:read`、`orgunit.orgunits:export`；其中 `object` 对齐 Casbin object/resource，`action` 对齐 Casbin action | 480/481 共享 |
| Capability Registry | 能力候选项与资源维度元数据；声明某资源是否需要组织范围，具体候选源与 options API 由 `DEV-PLAN-482` 承接 | 480/481/482 共享 |
| Role Assignment | 把角色授予 principal/team/position，并绑定组织范围 | 480 后续子计划 |
| Data Scope Binding | 授权记录中的组织范围，例如组织子树或全公司 | 480 后续子计划 |
| Authorization Decision | 运行时把主体、角色、范围合成裁决 | 480 |

### 3.2 角色定义归属

角色定义只维护功能权限。

示例：

| 资源 | 操作 | capability key | 角色页表现 |
| --- | --- | --- | --- |
| 组织管理 | 查看 | `orgunit.orgunits:read` | 勾选 |
| 组织管理 | 导出 | `orgunit.orgunits:export` | 勾选 |
| 员工信息管理 | 查看 | `employee.employees:read` | 勾选 |
| 员工信息管理 | 编辑 | `employee.employees:update` | 未勾选 |
| 薪酬管理 | 查看报表 | `salary.reports:read` | 勾选 |

角色页不显示“组织范围”“范围要求”“字段策略”。它只允许管理员确认这个角色能做哪些功能。

### 3.3 组织范围归属

组织范围只在用户授权/角色分配时绑定。系统根据所选角色包含的 capability，从 registry 推导是否需要组织范围。

示例：

| 已选角色能力 | registry 范围维度 | 用户授权时的配置 |
| --- | --- | --- |
| `orgunit.orgunits:read`、`orgunit.orgunits:export` | `organization` 必填 | 选择“鲜花事业部（含下级）” |
| `employee.employees:read` | `organization` 必填 | 复用同一个组织范围 |
| `iam.authz:read` | 无组织范围 | 不出现组织树选择 |

这避免角色定义里硬编码“鲜花事业部”，也避免让角色设计者为 `read/export` 逐项维护范围要求。

## 4. UI 设计方案

### 4.1 角色定义页

页面路径：`系统管理 > 角色管理 > 新建/编辑角色`

角色定义页的主区域只围绕一个角色展开：

| 区域 | 展示内容 | 交互 |
| --- | --- | --- |
| 角色基本信息 | 角色名称、角色标识、角色描述 | 输入并保存 |
| 功能权限分配 | 按资源分组展示操作，例如组织查看、组织导出、员工查看、薪酬查看 | 勾选/取消勾选 |
| 操作按钮 | `保存` | 保存即生效 |

角色定义页不得出现：

- 组织树或组织范围选择。
- 每个能力的“范围要求”开关。
- 字段策略、有效期、冲突检测或策略预览。
- “保存并发布”或“导出能力缺少范围要求”这类发布阻断。

### 4.2 用户授权页骨架

页面路径：`系统管理 > 用户管理 > 用户授权`

| 区域 | 展示内容 | 交互方式 |
| --- | --- | --- |
| 顶部用户选择器 | 当前被授权用户，例如 `user-b · 王小花` | 下拉选择用户，切换后加载该用户授权配置 |
| 主内容页签 | `角色`、`组织范围` | 点击页签切换同一用户的两类配置 |
| 保存动作 | `保存` | 保存即生效 |

页面不提供编辑角色定义本身的入口。管理员若要改角色功能权限，应回到角色定义页。

### 4.3 角色页签

角色页签是可加行表格。

| 列 | 展示内容 | 交互细节 |
| --- | --- | --- |
| 授权角色 | 角色下拉选择框，例如 `鲜花公司HR` | 支持添加行，每行选择一个角色 |
| 角色说明 | 只读说明该角色可以查看或操作什么，例如“查看组织、导出组织、查看员工与薪酬报表” | 随角色选择自动带出 |
| 操作 | `移除` | 删除当前角色行 |

### 4.4 组织范围页签

组织范围页签是可加行表格。

| 列 | 展示内容 | 交互细节 |
| --- | --- | --- |
| 组织 | 组织下拉或组织树选择结果，例如 `鲜花事业部` | 支持添加行，每行选择一个组织 |
| 包含下级组织 | 勾选框 | 初始为已选中；界面不额外显示说明文字 |
| 操作 | `移除` | 删除当前组织范围行 |

## 5. 保存与校验模型

本计划不直接建表。后续实现可选择文件策略、DB SoT 或混合保存模型，但必须满足：

1. 角色定义有稳定 `role_slug` 与修订号。
2. 能力 key 只能引用 `DEV-PLAN-482` registry 中的 `object/action`，序列化格式固定为 `object:action`；不得新增 `orgunit.view` 这类 `module.verb` 兼容别名。
3. 资源/能力 registry 能表达该资源在授权分配时是否需要组织范围。
4. 角色保存后立即生效，用户授权引用当前有效角色定义。
5. 用户授权保存时生成主体、角色行和组织范围行的可审计记录。
6. 授权保存缺少必填角色或组织范围时，在对应表格行内提示；不引入发布前完整性审查或多步骤 review。

## 6. ADR 摘要

- **决策 1：角色定义只表达功能权限**
  - **备选 A**：在角色定义中声明每个能力的范围要求。拒绝，会把功能权限和数据范围强耦合，并让 `read/export` 等操作重复配置。
  - **备选 B**：角色定义直接绑定“鲜花事业部”。拒绝，角色会变成某个组织实例的授权记录，无法复用。
  - **选定理由**：角色应回答“能做什么”；“在哪里做”属于授权分配事实。

- **决策 2：范围维度由资源/能力 registry 推导**
  - **备选 A**：管理员在角色设计页手工维护 `scope_required`。拒绝，容易遗漏、重复和阻断保存。
  - **备选 B**：完全不校验范围。拒绝，用户授权容易默认全租户。
  - **选定理由**：范围要求是资源维度元数据，系统在授权页自动提示和校验。

- **决策 3：用户授权界面用两个页签分开维护角色和组织范围**
  - **备选 A**：授权时同时选择字段策略、有效期、冲突检测和策略预览。拒绝，会让极简授权流程变成多职责配置页。
  - **备选 B**：添加角色弹窗里同时设置组织范围。拒绝，会让角色与范围再次耦合，也不利于批量维护多个角色或多个组织。
  - **选定理由**：当前闭环只需要“选用户 + 角色行 + 组织范围行”；其余能力后续按独立计划落地。

## 8. 风险与缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| 角色定义与用户授权混淆 | 管理员在角色页配置组织范围 | 分成角色管理和用户授权两个入口 |
| 范围要求回流角色页 | `scope_required` 被做成角色字段 | 范围维度归属 `object/action` registry，角色页只读功能权限 |
| 授权页过度设计 | 字段策略、有效期、冲突检测、策略预览回流 | 481 明确极简授权页只做用户选择、角色表格和组织范围表格 |
| 策略键漂移 | UI、policy、PDP、CubeBox 名称不一致 | registry + lint + 保存校验 |
| 默认全租户 | 授权时未选择范围仍保存 | 用户授权保存阶段校验必填组织范围 |
