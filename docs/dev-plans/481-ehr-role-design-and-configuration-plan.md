# DEV-PLAN-481：EHR 角色定义与极简用户授权方案

**状态**: 交互契约已冻结；487/489/489A 后端保存与运行时闭环已落地，角色定义/用户授权 UI 保存交互与 E2E 待实施（2026-05-02 CST）

## 0. 适用范围与评审分级

- **评审分级**：`T2`
- **范围一句话**：冻结角色定义与用户授权的最小交互边界；角色只定义功能权限，用户授权页顶部选择用户，主区用“角色 / 组织范围”两个页签维护授权表格，其中组织范围是首批用户授权闭环的一部分，不是未来占位。
- **关联模块/目录**：`pkg/authz/**`、`config/access/**`、`scripts/authz/**`、`internal/server/**`、`apps/web/src/**`
- **关联计划/标准**：`AGENTS.md`、`DEV-PLAN-000`、`DEV-PLAN-001`、`DEV-PLAN-012`、`DEV-PLAN-022`、`DEV-PLAN-032`、`DEV-PLAN-480`、`DEV-PLAN-480A`、`DEV-PLAN-482`、`DEV-PLAN-483`、`DEV-PLAN-487`、`DEV-PLAN-489`、`DEV-PLAN-489A`
- **用户入口/触点**：角色列表、角色详情、角色定义表单、用户角色分配、组织范围选择器

### 0.1 Simple > Easy 三问

1. **边界**：角色定义 owner 是基础信息和功能权限；用户授权 owner 是用户选择、角色行和组织范围行。字段安全、有效期、冲突检测、策略预览、授权摘要查询、保存审计解释不进入本轮极简授权界面。
2. **不变量**：角色定义页不出现组织范围，不要求管理员为每个 action 手工维护 `scope_required`；组织范围由资源/能力 registry 的系统元数据推导，并在用户授权页保存和校验。需要组织范围的角色授权缺少组织范围时不得默认全租户。
3. **可解释**：管理员先定义角色能做什么；进入用户授权页后先选择用户，再在角色页签添加角色行，在组织范围页签添加组织范围行并保存。可编辑页统一提供“取消 / 保存”动作；首期不再设置“保存并发布”、复制角色或发布阻断流程。

### 0.2 现状研究摘要

- `DEV-PLAN-480` 已冻结 EHR 授权体系蓝图，但早期草图把“范围要求”“字段策略”“完整性校验”放在角色设计页，容易把功能权限、数据范围和字段级安全强耦合。
- 当前 `config/access/policies/**` 与 Casbin 工具链只覆盖 anonymous/superadmin/bootstrap/static/system surface；普通 tenant 在线角色定义、范围绑定、保存工作流与运行时能力授权已由 `DEV-PLAN-487/489/489A` 后端闭环承接。481 当前剩余范围是把角色定义与用户授权 UI 保存交互接入这些服务端 SoT，并补齐 E2E。
- 更符合管理员心智的分层是：角色定义回答“能做什么”；用户授权回答“谁在什么组织范围内行使这个角色”。
- 本方案先冻结交互与契约；角色定义后端闭环由 `DEV-PLAN-487` 承接，组织范围 SoT、保存 API、运行时读取与强制裁剪由 `DEV-PLAN-489` 承接，多角色运行时 union、subject set 与审计由 `DEV-PLAN-489A` 承接。若实施需要新增 DB 表、迁移或在线写入实现，必须获得 schema 确认，不能用前端本地状态、prompt 或 Casbin object 字符串代替。

## 1. 背景与上下文

EHR 里的“角色”不是“给某个用户的一次授权记录”。角色应该表达一类职责的稳定功能权限模型，例如“鲜花公司 HR”“集团 HR”“组织只读用户”。

一个角色只回答：

- 角色名称、标识与描述是什么。
- 这个角色具备哪些功能权限，例如组织查看、组织维护、CubeBox 使用、字典查看。
- 这个角色是否可保存，是否存在无效 authz capability key、重复 key 或基础信息缺失。

以下内容不属于角色定义：

- “用户 B 能看鲜花事业部及下级”：这是用户授权/角色分配时绑定的数据范围。
- “组织资源需要组织范围”：这是资源/能力 registry 的系统元数据，不应让角色设计者逐项手工声明。
- “字段隐藏、脱敏、有效期、冲突检测、策略预览”：这些不是本轮极简授权界面的交付项。
- “复制一个已有角色再改”：这是批量建模便利功能，不进入首期角色定义闭环。

## 2. 目标与非目标

### 2.1 核心目标

1. [ ] 冻结职责边界：角色定义只包含基础信息和功能权限；用户授权只包含主体、角色引用和组织范围绑定。
2. [ ] 定义角色定义 UI：角色基本信息、资源-操作矩阵、统一“取消 / 保存”按钮，保存即生效。
3. [ ] 定义用户授权 UI：页面顶部选择用户并提供“取消 / 保存”按钮；主区拆成“角色”和“组织范围”两个页签；每个页签都是可加行表格。
4. [ ] 明确校验边界：角色定义保存 API、持久化和服务端校验由 `DEV-PLAN-487` 承接；用户授权的组织范围保存、必填校验和运行时生效必须作为首批用户授权闭环承接，不能只冻结页面结构。

### 2.2 非目标

1. 不在本计划正文中直接新增 DB schema、迁移、在线角色写入 API 或后台生效工作流；角色定义保存与运行时生效由 `DEV-PLAN-487` 承接，用户授权组织范围若要交付为可保存功能，必须由首批实施子计划冻结 SoT 并获得 schema 确认。
2. 不让角色定义页维护组织范围或 `scope_required`。
3. 不把字段策略硬编码在角色定义里，也不在用户授权页选择、编辑或预览字段安全配置。
4. 不在极简授权界面提供有效期、冲突检测、策略预览、多步骤流程或多维度并行设置。
5. 不替换 Casbin，不引入 OPA/CEL 或第二套 policy engine。
6. 不恢复 legacy、SetID、scope/package 或旧 org_level/scope_type/scope_key 语义。
7. 不实现角色复制、角色克隆、从已有角色新建，也不预留复制按钮、复制 API 或 clone payload；后续若需要必须另起计划定义 `role_slug` 唯一性、审计差异和不可复制字段。

### 2.3 用户可见性交付

- **用户可见入口**：
  - `系统管理 > 角色管理 > 新建/编辑角色`：只编辑角色基础信息和功能权限。
  - `系统管理 > 用户管理 > 用户授权`：选择用户后，在“角色”和“组织范围”两个页签中分别维护授权行。
- **最小可操作闭环**：
  - 管理员创建 `flower-hr`，勾选 `orgunit.orgunits:read`、`orgunit.orgunits:admin`、`cubebox.conversations:use`、`iam.dicts:read`。
  - 管理员在角色定义页编辑名称、slug、说明和 authz capability key 集合。
  - 管理员在用户授权页顶部选择用户 B。
  - 管理员在“角色”页签添加 `flower-hr` 行，角色说明列展示该角色可查看或操作的内容。
  - 管理员在“组织范围”页签添加“鲜花事业部”行，“包含下级组织”列初始为已选中，并保存为运行时数据范围事实。
  - 若 `flower-hr` 包含 `scope_dimension=organization` 的 authz capability 但未配置组织范围，保存必须失败，不能默认全租户。

## 3. 概念模型与边界

### 3.1 核心概念

| 概念 | 含义 | 归属 |
| --- | --- | --- |
| Role Definition | 角色定义：名称、slug、说明、能力集合、修订号 | 481 UI / 487 后端闭环 |
| Authz Capability | 稳定授权项标识，序列化为 `object:action`，例如 `orgunit.orgunits:read`、`orgunit.orgunits:admin`；其中 `object` 对齐 Casbin object/resource，`action` 对齐 Casbin action；它不是 API 地址 | 480/481 共享 |
| Authz Capability Registry | 授权项候选项与资源维度元数据；声明某资源是否需要组织范围，具体候选源与 options API 由 `DEV-PLAN-482` 承接 | 480/481/482 共享 |
| Role Assignment | 把一个或多个角色授予 principal，并绑定组织范围；首批运行时按 489A 做多角色 union | 481 UI / 480 运行时 / `DEV-PLAN-489A` |
| Data Scope Binding | 授权记录中的组织范围，例如组织子树或全公司；首批至少表达 `org root + include_descendants` | 481 UI / 480 运行时 / `DEV-PLAN-489` |
| Authorization Decision | 运行时把主体、角色、范围合成裁决 | 480 |

### 3.2 角色定义归属

角色定义只维护功能权限。

示例：

| 资源 | 操作 | 授权项标识 | 角色页表现 |
| --- | --- | --- | --- |
| 组织管理 | 查看 | `orgunit.orgunits:read` | 勾选 |
| 组织管理 | 维护 | `orgunit.orgunits:admin` | 勾选 |
| CubeBox 对话 | 使用 | `cubebox.conversations:use` | 勾选 |
| CubeBox 对话 | 查看会话 | `cubebox.conversations:read` | 未勾选 |
| IAM 字典 | 查看 | `iam.dicts:read` | 未勾选 |

角色页不显示“组织范围”“范围要求”“字段策略”。它只允许管理员确认这个角色能做哪些功能。

### 3.3 组织范围归属

组织范围只在用户授权/角色分配时绑定。系统根据所选角色包含的 authz capability，从 registry 推导是否需要组织范围。该页签不是未来占位；当用户授权功能进入可保存状态时，组织范围行必须随授权一同保存，并参与运行时裁决。

示例：

| 已选角色能力 | registry 范围维度 | 用户授权时的配置 |
| --- | --- | --- |
| `orgunit.orgunits:read`、`orgunit.orgunits:admin` | `organization` 必填 | 选择“鲜花事业部（含下级）” |
| `cubebox.conversations:use` | 无组织范围 | 不出现组织树选择 |
| `iam.dicts:read` | 无组织范围 | 不出现组织树选择 |

这避免角色定义里硬编码“鲜花事业部”，也避免让角色设计者为 `read/admin` 逐项维护范围要求。若同一用户的多个角色都需要 `organization` 范围，首批可复用同一组组织范围行；更复杂的“按角色分别设置范围”必须另起计划冻结，不在本轮极简 UI 中临时扩展。

`employee.employees:*`、`salary.reports:*` 与 `orgunit.orgunits:export` 暂不作为首期可实施 UI 项或角色候选项。只有当对应模块、tenant API 覆盖、registry entry 与 `DEV-PLAN-484` 覆盖门禁全部落地后，才允许进入 `assignable=true` 功能授权项。

## 4. UI 设计方案

### 4.1 角色定义页

页面路径：`系统管理 > 角色管理 > 新建/编辑角色`

角色定义页的主区域只围绕一个角色展开：

| 区域 | 展示内容 | 交互 |
| --- | --- | --- |
| 角色基本信息 | 角色名称、角色标识、角色描述 | 输入并保存 |
| 功能权限分配 | 按资源分组展示操作，例如组织查看、组织维护、CubeBox 使用、字典查看 | 勾选/取消勾选 |
| 操作按钮 | `取消`、`保存` | 不提供复制角色入口；保存 API 与持久化语义由 `DEV-PLAN-487` 冻结 |

角色定义页不得出现：

- 组织树或组织范围选择。
- 每个能力的“范围要求”开关。
- 字段策略、有效期、冲突检测或策略预览。
- “保存并发布”或“导出能力缺少范围要求”这类发布阻断。
- “复制角色”“克隆角色”“从此角色新建”一类复制入口。

### 4.2 用户授权页骨架

页面路径：`系统管理 > 用户管理 > 用户授权`

| 区域 | 展示内容 | 交互方式 |
| --- | --- | --- |
| 顶部用户选择器 | 当前被授权用户，例如 `user-b · 王小花` | 下拉选择用户，切换后加载该用户授权配置 |
| 主内容页签 | `角色`、`组织范围` | 点击页签切换同一用户的两类配置 |
| 操作按钮 | `取消`、`保存` | 保存必须覆盖角色行与组织范围行；不提供保存并发布 |

页面不提供编辑角色定义本身的入口。管理员若要改角色功能权限，应回到角色定义页。

### 4.3 角色页签

角色页签是可加行表格。

| 列 | 展示内容 | 交互细节 |
| --- | --- | --- |
| 授权角色 | 角色下拉选择框，例如 `鲜花公司HR` | 支持添加行，每行选择一个角色 |
| 角色说明 | 只读说明该角色可以查看或操作什么，例如“查看组织、维护组织、使用 CubeBox 与查看字典” | 随角色选择自动带出 |
| 操作 | `移除` | 删除当前角色行 |

### 4.4 组织范围页签

组织范围页签是可加行表格。

| 列 | 展示内容 | 交互细节 |
| --- | --- | --- |
| 组织 | 组织下拉或组织树选择结果，例如 `鲜花事业部` | 支持添加行，每行选择一个组织 |
| 包含下级组织 | 勾选框 | 初始为已选中；界面不额外显示说明文字 |
| 操作 | `移除` | 删除当前组织范围行 |

## 5. 保存与校验边界

本计划不直接建表，也不在正文中给出迁移 SQL。角色定义的在线保存 API、DB SoT、服务端校验与普通 tenant role 运行时能力授权来源由 `DEV-PLAN-487` 冻结；用户授权组织范围的持久化模型必须由 `DEV-PLAN-489` 冻结；多角色 union 运行时语义必须由 `DEV-PLAN-489A` 冻结。任何新增 DB schema 都必须先获得用户手工确认。后续实现不得选择前端本地状态、prompt 或 Casbin object 字符串代替服务端 SoT，且必须满足：

1. 角色定义有稳定 `role_slug` 与修订号；具体保存 payload、并发修订号和 system-managed 角色保护以 `DEV-PLAN-487` 为准。
2. authz capability key 只能引用 `DEV-PLAN-482` registry 中的 `object/action`，序列化格式固定为 `object:action`；按 `DEV-PLAN-483` 不接受 `orgunit.read`、`dict.admin`、`cubebox.conversations.read` 这类旧 key，也不得新增兼容别名。
3. 资源/能力 registry 能表达该资源在授权分配时是否需要组织范围。
4. 用户授权保存引用当前有效角色定义，并同时保存组织范围绑定。
5. 角色保存模型不接受 `source_role_slug`、`clone_from`、`copy_permissions_from` 等复制语义字段；角色新建必须显式提交基础信息和能力集合。
6. 用户授权保存时，若所选角色集合按 489A union 后包含 `scope_dimension=organization` 的 authz capability，则至少有一条组织范围绑定；缺失时返回明确错误，不得隐式写入全租户。
7. 用户授权保存后，运行时授权裁决必须能读取该组织范围事实；UI 保存成功但 API/CubeBox 仍按全租户执行视为未交付。

## 6. ADR 摘要

- **决策 1：角色定义只表达功能权限**
  - **备选 A**：在角色定义中声明每个能力的范围要求。拒绝，会把功能权限和数据范围强耦合，并让 `read/export` 等操作重复配置。
  - **备选 B**：角色定义直接绑定“鲜花事业部”。拒绝，角色会变成某个组织实例的授权记录，无法复用。
  - **选定理由**：角色应回答“能做什么”；“在哪里做”属于授权分配事实。

- **决策 2：范围维度由资源/能力 registry 推导**
  - **备选 A**：管理员在角色设计页手工维护 `scope_required`。拒绝，容易遗漏、重复和阻断保存。
  - **备选 B**：完全不校验范围。拒绝，用户授权容易默认全租户。
  - **选定理由**：范围要求是资源维度元数据，系统在授权页自动提示和校验。

- **决策 3：用户授权界面用两个页签分开维护角色和组织范围**
  - **备选 A**：授权时同时选择字段策略、有效期、冲突检测和策略预览。拒绝，会让极简授权流程变成多职责配置页。
  - **备选 B**：添加角色弹窗里同时设置组织范围。拒绝，会让角色与范围再次耦合，也不利于批量维护多个角色或多个组织。
  - **选定理由**：当前闭环只需要“选用户 + 角色行 + 组织范围行”；其余能力后续按独立计划落地。

## 8. 风险与缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| 角色定义与用户授权混淆 | 管理员在角色页配置组织范围 | 分成角色管理和用户授权两个入口 |
| 范围要求回流角色页 | `scope_required` 被做成角色字段 | 范围维度归属 `object/action` registry，角色页只读功能权限 |
| 授权页过度设计 | 字段策略、有效期、冲突检测、策略预览回流 | 481 明确极简授权页只做用户选择、角色表格和组织范围表格 |
| 策略键漂移 | UI、policy、PDP、CubeBox 名称不一致 | registry + lint + 保存校验 |
| 复制角色语义过早进入首期 | UI 出现复制按钮或 API 接收 clone 字段 | 首期删除复制入口；后续另起计划冻结唯一标识、审计差异和不可复制字段 |
| 默认全租户 | 授权时未选择范围仍保存 | 后续保存实现必须校验必填组织范围；本计划不冻结 UI 行内错误态 |
| UI 占位不生效 | 页面能选组织但保存后 API 仍按全租户执行 | 用户授权组织范围必须进入 SoT 与运行时裁决，否则不能作为可交付闭环 |
