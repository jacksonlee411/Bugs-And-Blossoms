# DEV-PLAN-431A：CubeBox 页面设计契约（承接 DEV-PLAN-431）

**状态**: 规划中（2026-04-20 08:27 CST）

## 0. 适用范围与评审分级

- **评审分级**：`T2`
- **范围一句话**：在 `DEV-PLAN-431` 已冻结的 UI 协议、状态机与右悬挂壳层基础上，单独冻结 CubeBox 的页面层视觉与交互合同，避免实现阶段把“协议层正确”再次做成“页面层漂移”。
- **关联模块/目录**：`docs/dev-plans/431-codex-ui-protocol-and-shell-reuse-plan.md`、`apps/web`、`designs/cubebox1.pen`
- **关联计划/标准**：`DEV-PLAN-002`、`DEV-PLAN-005`、`DEV-PLAN-010`、`DEV-PLAN-103`、`DEV-PLAN-431`

### 0.1 Simple > Easy 三问

1. **边界**：本计划只冻结 CubeBox 的页面设计语义、布局层级、视觉密度与控件命名；不替代 `DEV-PLAN-431` 的事件协议、turn/conversation 状态机与 reducer 设计。
2. **不变量**：页面层必须优先呈现“真实对话回复”而不是“方案说明板”；视觉上允许参考 Codex，但不得把 `codex` 当作 CubeBox 品牌文案固化在 UI 合同里。
3. **可解释**：reviewer 必须能在 5 分钟内说明 CubeBox 页面为何是“Codex 风格的阅读式对话界面”，以及底部模型 pill、composer、timeline、header 分别代表什么业务语义。

## 1. 背景与上下文

`DEV-PLAN-431` 已冻结 CubeBox 的 UI 协议、timeline reducer、右悬挂抽屉和会话恢复语义，但它并不直接冻结页面视觉层的正式合同。实际页面设计过程中已经暴露出一个典型漂移风险：

- 协议/状态机层虽然正确，但页面层容易被做成“分析面板”“dashboard 卡片”或“设计说明页”。
- 参考图显示的真实目标更接近 **单列、白底、正文主导、灰色 code pill、底部高圆角 composer** 的阅读式对话界面。
- 底部中间的 `codex` pill 在参考图里是“当前所选模型”的承载形态；对于 CubeBox，这里必须表达 **当前引用的大模型**，而不是产品名称或品牌标签。

因此需要把页面层合同单独抽出来，成为 `DEV-PLAN-431` 的页面面 SSOT。

## 2. 目标与非目标

### 2.1 核心目标

1. 冻结 CubeBox 主会话页的页面结构、视觉重心与控件语义。
2. 冻结底部模型 pill 的业务含义：显示当前引用的大模型，而非 `codex` 品牌。
3. 冻结“什么该出现在主页面、什么不该出现在主页面”的 stopline，阻止再次回到 dashboard/分析卡片形态。
4. 为 `apps/web` 的实现和 `designs/cubebox1.pen` 等设计稿提供页面层判断标准。

### 2.2 非目标

1. 不重新定义 `turn.started`、`turn.agent_message.delta`、`turn.completed` 等事件协议。
2. 不替代模型管理页、管理员配置页或 `DEV-PLAN-435` 的模型治理 UI。
3. 不冻结具体图标库实现细节；只冻结它们的语义位置与视觉权重。
4. 不把参考图逐像素临摹作为唯一目标；本计划强调的是页面合同与语义收敛。

### 2.3 用户可见性交付

- [ ] 用户从 Web Shell 进入 CubeBox 时，首屏能直接识别这是“对话界面”，而不是“架构介绍页”或“分析看板”。
- [ ] 用户能在不阅读额外说明文案的前提下理解：标题栏、正文回复区、底部输入区、模型 pill、发送按钮各自的作用。
- [ ] 当前模型名称在底部中间位置可见，且文案来自真实模型配置，而不是写死的 `codex`。

## 3. 页面设计契约

### 3.1 页面层级（冻结）

CubeBox 主页面必须保持以下视觉层级：

1. **顶栏**：返回/标题/轻量工具图标。
2. **正文回复区**：单列长文本输出，是绝对主角。
3. **底部 composer**：高圆角输入区，固定在页面底部感知位置。

禁止把以下区域做成和正文等权重的主区域：

- dashboard 卡片区
- 三栏信息概览区
- 架构图/说明板区
- 功能矩阵或“采纳/裁剪/重构”对照区

### 3.2 正文区合同（冻结）

正文区必须遵循：

- 默认形态是 **真实 assistant 回复**，不是页面设计说明。
- 以自然语言长段输出为主，可带项目符号、少量二级子项。
- 允许灰色 code pill 嵌入在句子中，用于表示分支名、commit、对象 ID、模型名等“短且需要强调”的机器可读标识。
- 不允许在正文主区放置成排的功能标签作为主要视觉元素。
- 不允许用彩色信息卡片替代长文回复。

### 3.3 顶栏合同（冻结）

- 顶栏为轻量级辅助区域，不得抢正文权重。
- 标题应使用当前会话标题或话题标题，可截断，但必须保留“这是一条会话”的语义。
- 顶栏图标只允许承担轻量行为，并冻结为三项：历史记录、设置、新聊天。
- 右上角第一个图标的正式语义冻结为：**历史对话记录入口**。点击后应打开历史会话列表/最近任务列表，而不是执行刷新、重新生成或重新加载当前会话。
- 右上角第二个图标的正式语义冻结为：**设置入口**。点击后应打开 CubeBox 设置面板或跳转设置页。
- 右上角第三个图标的正式语义冻结为：**新聊天入口**。点击后应创建新会话或清空当前草稿进入新会话态。
- 不得在顶栏中直接放置模型治理、插件市场、shell/patch/exec 等高风险入口。

### 3.4 底部 composer 合同（冻结）

- composer 必须是页面底部最强的第二视觉区域，仅次于正文。
- 默认由三部分构成：
  - 输入占位/草稿区
  - 底栏工具区
  - 发送按钮
- 输入区必须足够高，允许表达“可输入多行内容”的预期；不应退化为单行细输入框。
- 发送按钮必须始终位于右下角稳定位置。

### 3.5 模型 pill 合同（新增冻结）

底部中间的模型 pill 不是产品品牌，不是页面标题，也不是固定文案。它的正式语义是：

- **当前本次对话所引用的大模型**

因此：

- UI 上显示的文本必须来自当前选中的模型标识或其用户可读别名。
- 开发/设计阶段若用占位文案，允许临时写为 `codex`、`gpt-5.4`、`claude-sonnet-4` 等，但实现前必须改为真实绑定。
- 不得把该 pill 写死为 `codex` 并长期保留在 CubeBox 正式页面中。
- 不得把这个 pill 解释为 CubeBox 品牌或子产品名称。

### 3.6 不需要的控件（冻结）

对 CubeBox 主会话页，以下控件默认不需要直接出现：

- “完全访问权限”这类 Codex/agent 权限提示控件
- shell/patch/file/plugin/marketplace 入口
- 面向开发代理的审批级别切换器

若后续确有新增需要，必须在新计划文档中明确其业务语义、入口必要性和 stopline。

## 4. 视觉风格约束

### 4.1 主风格

- 白底/浅暖灰底
- 黑灰正文
- 浅灰 code pill
- 弱装饰、强内容
- 大圆角底部 composer

### 4.2 允许参考 Codex 的部分

- 单列阅读式会话结构
- 灰色 code pill 的视觉语法
- 底部高圆角 composer 的信息架构
- 正文区的 Markdown 长回复气质

### 4.3 不允许直接照搬的部分

- `codex` 品牌字样作为正式产品文案
- ChatGPT 账号、plan、rate limit 等顶栏或状态元素
- Agent 权限控制语义、插件市场、shell/file/patch/exec 操作表述

## 5. 页面元素语义表

| 页面元素 | 位置 | 正式语义 | 禁止误用 |
| --- | --- | --- | --- |
| 会话标题 | 顶栏左中 | 当前对话标题/话题 | 页面设计方案标题、架构文档标题 |
| 正文回复区 | 页面主区 | assistant 长回复 / timeline 内容 | 设计说明板、dashboard 卡片组 |
| code pill | 正文内或底栏 | 短标识强调（分支、commit、模型名等） | 成排功能标签、品牌徽标 |
| 模型 pill | 底栏中间 | 当前所引用的大模型 | `codex` 品牌固定文案 |
| 发送按钮 | 底栏右侧 | 发送当前输入 | 运行 shell / 执行 patch |

## 6. 实施约束与切片

1. [ ] 在 `designs/cubebox1.pen` 中把底部 `codex` pill 视为“当前模型 pill”占位，不再作为品牌语义解释。
2. [ ] 在 `apps/web` 中实现主会话页时，把模型 pill 绑定到真实模型选择状态。
3. [ ] 将页面层实现与 `DEV-PLAN-431` 的 conversation/timeline/composer/status bar 对齐。
4. [ ] 如新增空状态、恢复状态、流式状态稿，必须继承本计划的页面层合同，不得回流到卡片式说明页。

## 7. 测试与验收

- [ ] 设计稿验收：主页面首屏的视觉主角必须是正文回复区，而不是说明卡片。
- [ ] 语义验收：底部模型 pill 文案必须能解释为“当前模型”，而非“CubeBox 子品牌”。
- [ ] UI 验收：移除“完全访问权限”一类与 CubeBox 主会话页无关的控件。
- [ ] 响应式验收：宽度变化后，正文仍保持单列阅读结构，composer 仍保持底部主输入区地位。

## 8. Stopline

- 不得把 CubeBox 主会话页做成 dashboard、分析板、卡片矩阵或架构海报。
- 不得把底部模型 pill 固化为 `codex` 品牌文案。
- 不得重新引入“完全访问权限”等不属于 CubeBox 页面语义的控件。
- 不得在正文区大面积铺排功能标签或说明卡片，取代真实回复内容。
