# DEV-PLAN-460：CubeBox 数字助手定位与执行契约

**状态**: 规划中（2026-04-23 09:46 CST）

## 0. 适用范围与评审分级

- **评审分级**：`T2`

- **关联计划/标准**：`AGENTS.md`、`DEV-PLAN-003`、`DEV-PLAN-017`、`DEV-PLAN-019`、`DEV-PLAN-022`、`DEV-PLAN-430`、`DEV-PLAN-433`、`DEV-PLAN-434`、`DEV-PLAN-435`、`DEV-PLAN-437A`、`DEV-PLAN-438`
- **用户入口/触点**：Web Shell 右侧 `CubeBox` 抽屉、`/internal/cubebox/**`、模型配置管理面、后续业务模块内可被 `CubeBox` 调用的查询/操作 API 与脚本

### 0.1 Simple > Easy 三问

1. **边界**：`CubeBox` 是当前用户调用的数字助手入口，不是独立用户、独立角色、独立授权主体，也不是第二套业务运行面。
2. **不变量**：所有查询、操作、EHR 写入都必须继承当前用户、当前租户、当前 session、当前会话上下文；任何业务写入都必须回到现有模块 One Door、事务、RLS、Authz 与审计链路。
3. **可解释**：reviewer 必须能在 5 分钟内说明 `CubeBox` 的权限从哪里来、它如何把用户意图转成查询/操作、文档在编排里扮演什么角色，以及它为什么不是权限提升通道。

### 0.2 现状研究摘要

- **现状实现**：`CubeBox` 已具备右侧抽屉 UI、会话持久化/恢复、流式回复、provider 网关、模型设置、canonical context reinjection、manual/pre-turn compaction 与 `/internal/cubebox/**` API 主链。
- **现状约束**：服务端能力判定与会话存储已按 `tenant + principal` 隔离；模型调用统一走服务端网关；当前主链仍以会话、上下文和 provider 调用为核心，通用业务工具编排与文档接入尚未作为完整运行时封板。
- **最容易出错的位置**：把 `CubeBox` 误做成独立授权主体；把“模型能回答”误当成“已经具备业务执行闭环”；让前端自行拼业务 prompt；让文档知识与真实权限边界脱节。
- **本次不沿用的“容易做法”**：不把 `CubeBox` 做成 service account；不走后门脚本；不让模型输出直接代替业务提交；不把租户专用文档读取做成绕过现有权限的知识回流通道。

## 1. 背景与上下文

当前 `430` 系列已冻结 `CubeBox` 的 UI、网关、会话、compaction 与管理面方向，但“它究竟在产品上是什么”仍分散在多个子计划里。用户现已明确要求把 `CubeBox` 定义为用户的数字助手，并冻结三条产品原则：

1. 它的权限完全来自当前操作它的用户，而不是另外授权。
2. 它通过调用 API 和脚本完成用户要求的查询、操作和 EHR。
3. 它通过阅读本项目通用与租户专用文档、API 文档等，借助大模型把用户意图自动编排成一系列系统查询和操作动作。
4. 它在写入场景中只是现有业务写路径的提案器和确认器，不是新的写入执行主体；确认后仍由当前用户身份调用现有业务 API，最终复用现有模块的 One Door 写链路。

本计划的目标不是重写 `430`，而是把上述三点上升为所有 `CubeBox` 后续实现共同遵守的上位契约，避免后续在业务工具接入、文档接入、权限裁决和写入链路上发生语义漂移。

## 2. 目标与非目标

### 2.1 核心目标

- [ ] 冻结 `CubeBox` 的产品定位：它是“当前用户的数字助手”，不是独立代理主体。
- [ ] 冻结权限与执行边界：查询/操作/EHR 写入完全继承当前用户权限，写入必须显式确认。
- [ ] 冻结文档驱动编排语义：文档是意图理解与动作规划依据，不是授权来源。
- [ ] 冻结后续业务工具接入与审计归属的评审口径，避免出现第二套授权链或第二写入口。

### 2.2 非目标

- 不在本计划内新增数据库表、路由或运行时代码。
- 不把 `CubeBox` 升级为通用 agent 平台、工作流平台、PDP、策略引擎或独立审批系统。
- 不在本计划内实现完整的“租户专用文档仓库 + 文档检索基础设施 + tool registry”。
- 不改变 `430/433/434/435/437A` 已冻结的现有 owner 边界。

### 2.3 用户可见性交付

- **用户可见入口**：仍以现有右侧抽屉 `CubeBox` 入口为主。
- **最小可操作闭环**：用户在现有抽屉里发起问题，`CubeBox` 用当前用户权限完成查询/建议/后续操作编排，并在需要写入时先向用户确认动作与关键字段。
- **本计划交付形态**：文档契约 SSOT；后续业务与工具接入按本契约验收，避免“能聊天但不能正确代表用户”的伪闭环。

## 3. 核心定义

### 3.1 CubeBox 的正式定位

`CubeBox` 是内嵌在本项目 Web Shell 中的数字助手执行入口。它的职责是：

1. 理解当前用户输入的意图。
2. 结合当前页面、当前业务对象、当前用户权限、当前租户上下文和相关文档，推导出可执行的查询、分析或操作步骤。
3. 通过现有 API、脚本和后续受控工具能力完成这些步骤。
4. 在写入场景中，把自然语言请求转换为贴近现有业务 API 的结构化提案，并请求用户确认。
5. 将结果、过程状态、错误和待确认动作反馈给用户。

`CubeBox` 不是：

- 独立账号
- 独立授权主体
- 独立审批人
- 脱离现有业务系统的第二操作后台
- 可以越权读取或写入数据的 service account
- 新的业务写入执行主体

### 3.2 权限来源的正式定义

`CubeBox` 的权限完全来自当前正在操作它的用户。

具体冻结如下：

1. `CubeBox` 本身不拥有独立 `subject`。
2. 所有能力检查都必须映射回当前用户在当前租户内已有的 `subject/domain/object/action`。
3. 文档、模型、脚本、工具、上下文缓存都不能成为额外授权来源。
4. 若当前用户无权读取某对象、执行某操作或查看某配置，`CubeBox` 也无权代为执行。
5. 审计中的 `actor/principal` 必须仍是当前用户；`CubeBox` 只能作为 `channel/source/tool` 记录。

该定义承接并强化 `DEV-PLAN-430` 已冻结的原则，不得在后续实现中偏离为代理授权、独立主体或权限提升模型。

### 3.3 文档在 CubeBox 中的角色定义

文档是 `CubeBox` 的“理解与编排依据”，不是“授权依据”。

文档可用于：

- 帮助模型理解业务术语、流程、字段语义、约束和操作顺序
- 帮助模型识别“应该调用哪些 API / 脚本 / 系统能力”
- 帮助模型生成更稳定的查询计划、操作计划和解释输出
- 帮助模型区分通用规则与租户专用规则

文档不可用于：

- 赋予模型额外权限
- 绕过系统已有的 Authz / RLS / One Door
- 代替后端真实能力检查
- 代替业务 API 的参数校验、事务和错误映射

## 4. 执行模型

### 4.1 意图到动作的主流程

`CubeBox` 的最小主流程只冻结到原则级，不冻结未来 planner/runtime 的内部步骤：

1. 接收当前用户输入。
2. 结合当前会话、当前页面、当前业务对象、当前用户权限和相关文档，形成受控的查询或操作意图。
3. 通过现有 API、脚本或后续受控工具能力执行该意图。
4. 若涉及正式业务写入，先把意图收敛为贴近现有业务 API 的结构化提案，并向用户展示待执行动作、关键字段、影响对象与预期结果。
5. 用户确认后，仍按当前用户身份调用现有业务 API 或内核入口，复用现有模块 One Door 写链路，而不是经由 `CubeBox` 形成新的提交主链。
6. 将结果返回给用户。

本节不预设未来必须存在某种固定的 `planner`、`action chain`、`executor` 或等价运行时结构；后续若需要引入具体编排器，由对应实施计划另行冻结。

### 4.2 查询、操作与 EHR 的正式边界

本项目中的“查询、操作和 EHR”在 `CubeBox` 语义下统一解释为：

- **查询**：读取用户已有权限可见的业务数据、配置数据、审计摘要或系统状态。
- **操作**：不改变业务事实源的受控动作，例如验证、试算、预检查、生成建议、编制参数。
- **EHR 写入**：通过现有业务模块 One Door 入口提交的正式业务变更。

冻结规则：

1. 查询可直接按当前用户权限执行。
2. 非写入型操作可在当前用户权限允许范围内直接执行。
3. 任何正式业务写入都必须经过用户确认。
4. `CubeBox` 在写入场景中的职责是“提案 + 确认”，不是新的写入执行主体。
5. 确认后提交时，仍必须以当前用户身份调用现有业务 API 或内核入口。
6. 不允许模型输出绕过业务 API 直接写库、直接改投影、直接跳过事务/RLS/Authz。

### 4.3 API 与脚本的正式准入

`CubeBox` 可以调用 API 和脚本，但这里只冻结最小准入原则，不提前扩张为工具治理框架。

准入原则冻结如下：

1. API 或脚本的执行权限必须能够回溯到当前用户，而不是 `CubeBox` 自身。
2. API 或脚本不得形成旁路写入口，不得绕过现有业务边界。

更细的工具注册、输入输出契约、失败语义和运行时组织方式，后移到对应专项计划。

### 4.4 文档驱动编排的正式定义

“文档驱动编排”在本仓中的正式含义是：

- 由文档解释“系统里有什么对象、什么规则、什么流程、应该如何调用”
- 由模型把用户自然语言请求转换成受控的系统动作计划
- 由后端和工具链负责真正执行、校验、失败处理和审计

因此：

- 文档负责“帮助理解与规划”
- 工具负责“实际执行”
- 权限系统负责“是否允许”
- 业务模块负责“如何正确落库”

四者不得混淆。

## 5. 权限与审计契约

### 5.1 权限继承不变量

- `CubeBox` 不得拥有独立 `subject`
- `CubeBox` 不得使用 service account
- `CubeBox` 不得把“能看到某份文档”解释为“能执行某个动作”
- `CubeBox` 不得在当前用户无权限时继续执行该动作
- `CubeBox` 不得缓存并复用已失效的旧权限上下文

### 5.2 写入确认不变量

- 写入前必须明确动作名称、影响对象、关键字段和预期结果
- 用户确认是提交前置条件，不得省略
- `CubeBox` 在写入场景中只负责生成结构化提案并发起确认，不负责成为新的写入执行主体
- 用户确认不替代后端权限校验、参数校验和业务不变量校验
- 用户确认后的执行 actor 仍是当前用户，且提交链路仍是现有业务 API / One Door

### 5.3 审计归属不变量

- 审计 `actor/principal` = 当前用户
- 审计 `tenant` = 当前租户
- 审计附加元数据可记录 `channel=cubebox`
- 可记录 `conversation_id`、`trace_id`
- 不得把 `CubeBox` 记录为业务授权主体

## 7. 后续实现触发器

当后续工作命中以下任一场景时，必须显式回链本计划：

1. 新增 `CubeBox` 可调用的业务 API 或脚本
2. 引入“项目通用文档 / 租户专用文档 / API 文档”给 `CubeBox` 消费
3. 引入新的工具编排运行时或执行框架
4. 允许 `CubeBox` 发起正式业务写入
5. 变更 `CubeBox` 的审计字段、权限边界或执行 actor 语义

## 8. 验收标准

- [ ] 后续实现不得把 `CubeBox` 设计成独立授权主体。
- [ ] 后续实现不得绕过当前用户权限、当前租户和当前 session。
- [ ] 文档接入能力只能增强意图理解与动作规划，不能替代权限校验。
- [ ] 任何业务写入都必须要求用户显式确认，并回到现有 One Door。
- [ ] `CubeBox` 在写入场景中只能充当“提案器和确认器”，不能成为新的写入执行主体。
- [ ] 审计里 `actor/principal` 仍是当前用户，`CubeBox` 只作为 `channel/source/tool`。
- [ ] 新的业务工具、脚本或 planner 方案在 dev-plan 中能清楚说明其权限来源、执行边界、失败路径与 owner。

## 9. 反模式与禁止项

- 不得把 `CubeBox` 做成 service account 或后台超管
- 不得在文档里写了某流程，就默认 `CubeBox` 有权执行该流程
- 不得让前端或模型直接拼装并提交未受控的业务写入
- 不得让 `CubeBox` 成为新的业务写入主入口或新的写入执行主体
- 不得新增第二写入口或绕过现有模块写边界
- 不得以“数字助手更智能”为理由扩大权限面
- 不得把租户专用文档接入做成跨租户知识泄漏通道
