## 1. 背景与上下文
1. [ ] `DEV-PLAN-246` 及其子计划 `241~245` 已把 Assistant 的知识资产、路由、澄清、回复与版本审计主链落地，并通过了现有运行态与端口级验证。
2. [ ] 但真实对话体验暴露出一个更严重的问题：系统在工程上变得更可控、更可审计，却在产品上变得更僵硬、更保守、更容易误导用户。
3. [ ] 当前用户反馈的典型症状包括：
   - [ ] 自然表达轻微变体即掉入 `uncertain` / `non_business_route`；
   - [ ] 输入中某一槽位格式失败时，其它本可继续推断/检索的槽位也被一并冻结；
   - [ ] 回复文本虽然“无技术泄露”，却偏离当前业务域，出现泛化 HR 菜单式话术；
   - [ ] 系统更像“事务型状态机入口”而不是“能帮助用户把事办成的对话助手”。
4. [ ] 现有 `246A/246B` 的通过，证明的是“结构化主链存在且 fail-closed 生效”，并不等价于“对话能力自然、灵活、低摩擦”。

## 2. 目标与非目标
1. [ ] 核心目标：对 `246` 系列实施结果做一次系统与架构层面的复盘，明确“为何工程成功但体验失败”。
2. [ ] 核心目标：冻结后续纠偏方向，避免继续在 `241~245` 层面做局部补丁式修修补补。
3. [ ] 核心目标：把后续改造重心从“每层各自更严谨”收敛为“由外部大模型承担单一会话语义核，本地 runtime 只负责上下文、检索、边界和执行”。
4. [ ] 核心目标：定义一套新的体验验收口径，使后续 Assistant 改造不再只用 `phase/route/reply_http` 判定成功。
5. [ ] 核心目标：所有后续实施仍必须遵守 Greenfield 不变量，尤其是：
   - [ ] 写入链路仍保持 One Door；
   - [ ] `confirm/commit` 仍由严格 gate 保护；
   - [ ] 不引入 legacy/双链路/第二写入口。
6. [ ] 核心目标：明确哪些本地层应该被删除或降级为审计投影，防止“外部模型已能承担语义，但本地多层状态机仍继续抢判断权”。
7. [ ] 非目标：本计划不直接完成 runtime 代码修复；代码实施应由后续子计划承接。
8. [ ] 非目标：本计划不引入新的数据库表、外部缓存、向量检索平台、MCP 第二主链或额外执行入口。

## 3. 现状判断：为什么 246 系列“工程完成”但“对话失败”
1. [ ] 当前架构把 Assistant 主要建模为：`intent -> route -> clarification -> phase -> reply` 的多层受控状态机，而不是“围绕用户目标持续修复上下文”的会话系统。
2. [ ] `241~245` 的拆分边界是按技术子系统划分的：知识资产、路由、澄清、理解治理、表达治理分别成立；但用户任务闭环缺少单一语义主核，导致语义在层间被重复投影与损耗。
3. [ ] 结果是：
   - [ ] 局部都“正确”；
   - [ ] 整体却不“顺滑”；
   - [ ] fail-closed 从高风险提交边界扩散到了低风险理解边界。
4. [ ] 更关键的是，这套结构默认“本地 runtime 自己也要拥有完整理解能力”，于是模型判断、本地路由、本地澄清、本地回复又形成四套并行判断权。
5. [ ] `246B` 的通过主要证明：
   - [ ] route/phase/gate/reply 等结构化字段存在；
   - [ ] 关键门禁阻断生效；
   - [ ] 文本不泄露技术信号；
   - [ ] 但没有充分证明：用户是否更少轮次完成任务、是否减少误导性提示、是否明显感到“被理解”。

## 4. 系统级根因拆解
### 4.1 理解层：仍是“脆弱解码”，不是“鲁棒理解”
1. [ ] 当前理解主链仍严重依赖最小词表、正则与 `plan_only` 兜底；知识资产解决了治理问题，但没有实质解决自然语言变体的吸收能力。
2. [ ] 这意味着：
   - [ ] `新建`、`创建`、`设立` 等轻微表达变体会产生结构性差异；
   - [ ] `YYYY/MM/DD`、`YYYY-MM-DD`、`今天`、`本周一` 等时间表达不在统一归一化层收敛；
   - [ ] “公司/组织/集团/总部”等业务同义词没有架构位置。
3. [ ] 根本问题不是模型能力不足，而是 runtime 试图用本地最小实现替代模型承担语义工作，却又没有建立真正足够强的归一化与修复能力，导致理解失败过早下沉为路由失败。

### 4.2 路由层：过度优先防误触发，牺牲了业务可恢复性
1. [ ] 当前路由设计的中心思想是“宁可不放行，也不误入动作链”。这对提交安全是合理的，但被提前应用到了理解阶段。
2. [ ] `plan_only` 一旦出现，系统倾向跳过 `business_action` 的恢复机会，直接转入 `knowledge_qa/chitchat/uncertain` 的非动作分流。
3. [ ] 这让系统从“谨慎”滑向“僵硬”：
   - [ ] 用户原本在表达业务意图；
   - [ ] 系统却先把它判成非业务或待澄清；
   - [ ] 随后 reply 再把这个错误分流包装成看似合理的话术。

### 4.3 澄清层：更像表单修复器，而不是会话修复器
1. [ ] 当前澄清主链主要围绕 `validation_errors`、缺字段、候选确认等信号工作，本质是在修复一个接近表单的数据结构。
2. [ ] 这样做在“缺字段”和“候选确认”场景上是有效的，但对“用户说得像业务、系统却未稳判动作”场景不够自然。
3. [ ] 结果是系统擅长问“还缺什么字段”，却不擅长问“你真正想做什么，我帮你继续收敛”。

### 4.4 数据检索层：检索被前置校验硬耦合，放大误导性反馈
1. [ ] 组织候选解析被绑定到“其它字段已先通过”的前提上；一旦日期等槽位校验失败，父组织检索可能根本不执行。
2. [ ] 但对用户呈现时，系统又可能同时给出“上级组织未匹配”的提示，形成错误归因。
3. [ ] 这属于系统设计问题，不是单点文案问题：检索和解释没有分开建模“未执行 / 执行失败 / 无结果 / 多结果”四种状态。

### 4.5 表达层：知识化了模板，却没有收住最终话语权
1. [ ] `245` 把 `Reply Guidance Pack + Realizer` 落地后，解决了 reply 资产化与版本审计问题，但最终文本仍经过模型改写。
2. [ ] 当 reply 模型使用的是宽泛“企业 HR 助手”人格，而后端实际只支持窄域组织变更时，就会出现领域漂移：
   - [ ] 后端在做 orgunit；
   - [ ] 文本却在谈请假、报销、入离职、薪资等泛 HR 场景。
3. [ ] 这说明 reply 层当前仍是“受控 fallback + 模型二次措辞”，而不是“领域受限表达器”。

### 4.6 架构层：重复判断权导致天然慢、硬、重复
1. [ ] 当前每轮消息往往要经过 `resolveIntent -> routeDecision -> clarificationDecision -> replyKind` 多次决策，且各阶段都可能重写前一阶段结果。
2. [ ] 这使系统出现三个系统性后果：
   - [ ] 时延被多次本地判断与多次模型调用放大；
   - [ ] 同一事实在不同层出现不同解释，增加冲突与误导；
   - [ ] 任一层保守降级，都会把整体会话拖入僵硬路径。
3. [ ] 换句话说，当前问题不是“某一层做错”，而是“不该有这么多层同时保留判断权”。

### 4.7 验收层：测到了结构正确，没测到体验正确
1. [ ] `246B` 的多数 PASS 指标是 `phase=...`、`route_kind=...`、`reply_http=200`、`text_len>0` 这类结构化证据。
2. [ ] 这些指标足以证明系统不会乱写、不会裸奔、不会越权，但不足以证明：
   - [ ] 文本是否忠实当前业务域；
   - [ ] 追问是否最少轮次；
   - [ ] 错误归因是否准确；
   - [ ] 用户是否明显感受到“更好办事”。

## 5. 架构层反思
1. [ ] 我们把“知识治理”误当成了“对话能力建设”的近似替代。事实证明，资产可编译、可审计、可冻结，并不会自动带来语言弹性。
2. [ ] 我们把“主源唯一”理解成“每一层都各自建立唯一真相”，结果造成了 `Intent`、`RouteDecision`、`ClarificationDecision`、`Phase`、`ReplyKind` 多层并行语义中心。
3. [ ] 我们低估了外部大模型已经足以承担窄域语义理解与下一问生成的事实，仍试图在本地 runtime 里重建一套缩小版对话大脑。
4. [ ] 我们把“最小实现”使用得过久：每一层都以最小版本进入主链，单层合理，叠加后形成整体体验断裂。
5. [ ] 我们让 fail-closed 口径覆盖到了低风险的语言理解边界，而不是把严格阻断集中在真正危险的 `confirm/commit` 边界。
6. [ ] 我们让本地系统过多回答“用户到底想干什么”，却没有把本地系统更明确地收缩为“为模型准备事实，并守住执行边界”。
7. [ ] 我们的架构更擅长回答“系统现在处于什么阶段”，却不擅长回答“为了让用户把事办成，下一步最应该做什么”。

## 6. 纠偏原则（冻结为后续实施母法）
1. [ ] **原则 A：外部大模型承担单一会话语义核。**
   - [ ] 用户意图理解、上下文修复、下一问生成、候选检索请求、用户可见回复，应优先由单一外部模型在一个受控 JSON 契约内完成；
   - [ ] 本地 runtime 不再保留与其平行的本地“第二大脑”。
2. [ ] **原则 B：本地 runtime 只负责事实、上下文、边界与执行。**
   - [ ] 本地职责冻结为：`Context Assembler`、`Readonly Resolver`、`Dry Run`、`Action Gate`、`Confirm Gate`、`Commit Adapter`、`Audit`；
   - [ ] 本地不再对用户目标进行多层重复分类。
3. [ ] **原则 C：删掉多余语义层，而不是继续修补它们。**
   - [ ] `intent -> route -> clarification -> reply` 不应继续作为四段主决策链长期存在；
   - [ ] `phase/reason_codes/reply_kind` 应降级为审计、观测或 UI 投影，不再主导理解。
4. [ ] **原则 D：禁止双脑与本地兜底理解回潮。**
   - [ ] 不得继续依赖本地 fallback extractor、local upgrade、`plan_only` 伪动作去修补模型理解；
   - [ ] 外部模型失败时应显式进入人工接管/明确失败，不得悄悄切回旧本地理解链。
5. [ ] **原则 E：检索状态与用户提示必须一一对应。**
   - [ ] 明确区分“未执行”“无匹配”“多候选”“唯一命中”“因边界校验暂缓执行”；
   - [ ] 禁止错误归因文案。
6. [ ] **原则 F：每轮模型调用预算必须受控。**
   - [ ] 默认每轮最多两次模型调用：一次语义决策，必要时一次基于检索结果的收敛；
   - [ ] 禁止再追加独立的 reply 润色调用作为常态主链。
7. [ ] **原则 G：fail-closed 重点保护提交，不主导理解。**
   - [ ] `confirm/commit` 继续严格；
   - [ ] `understand/ask/retrieve` 以“可恢复优先、误导最少、轮次最少”为目标。
8. [ ] **原则 H：验收标准从‘结构正确’升级到‘体验正确’。**
   - [ ] 以后续 dev-plan 明确新增体验型 case 与 stopline。

## 8. 验收与停止线
1. [ ] 若后续改造仍主要围绕新增关键词、正则、本地 router 分支、局部 prompt 文案补丁展开，而没有建立“外部模型单一语义核 + 本地最小职责”，则本纠偏失败。
2. [ ] 若本地 runtime 继续保留 `intent/route/clarification/reply` 四套并行判断权作为主链，而不是删除或降级为投影，则本纠偏失败。
3. [ ] 若模型失败后系统悄悄回退到旧本地理解链，而不是显式失败/人工接管，则本纠偏失败。
4. [ ] 若 `uncertain/non_business_route` 的比例下降了，但只是通过更激进放行动作链实现，而非真实提升理解与修复能力，则本纠偏失败。
5. [ ] 若 reply 继续作为独立模型改写层长期存在，或继续允许在 orgunit 会话中生成泛 HR 分类菜单式话术，则本纠偏失败。
6. [ ] 若候选检索仍无法区分“未执行”和“无匹配”，则本纠偏失败。
7. [ ] 若后续主链默认每轮超过两次模型调用，且没有明确证据证明收益大于复杂度与时延成本，则本纠偏失败。
8. [ ] 若体验验收仍仅使用 `phase/route/reply_http/text_len` 等结构性指标，则本纠偏失败。
9. [ ] 若后续实施引入 legacy、双链路或第二写入口来“掩盖僵硬”，则本纠偏失败。
