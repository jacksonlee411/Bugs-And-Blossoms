# [Archived] DEV-PLAN-095：MUI X 升级子计划 P4（稳定化与性能压测）

**状态**: 已归档（2026-02-22，范围收口后不再单列实施）

> 本计划承接 `DEV-PLAN-090` 的 §5.5（Phase 4），聚焦性能基线、端到端稳定性和最终验收报告。

## 1. 子计划范围

- 对应 `DEV-PLAN-090` 步骤：14~16。
- 时间窗口：1~2 周。
- 输出形态：可复现性能报告 + 稳定性报告 + 升级验收报告。

## 2. 核心目标（DoD）

- [ ] 完成 10k/50k 数据量的滚动、筛选、切页性能压测。
- [ ] 完成关键链路 E2E 回归并冻结可追溯快照。
- [ ] 形成升级完成报告（功能覆盖、性能结果、门禁状态、遗留风险）。

## 3. 非目标

- 不在本阶段继续新增新页面。
- 不在本阶段扩展新的 UI 技术栈能力（只做稳定化）。

## 4. 实施步骤

1. [ ] 建立性能基线场景
   - 场景 A：大数据量分页 + 表头筛选。
   - 场景 B：树-表联动 + 详情侧栏切换。
   - 场景 C：批量选择 + 批量操作确认链路。
2. [ ] 定义并执行压测
   - 指标：首屏时间、交互延迟、长任务占比、内存峰值。
   - 对比：迁移前基线 vs 迁移后结果。
3. [ ] E2E 稳定化
   - 核心业务流覆盖：组织、人员、任职、审批。
   - 失败用例必须归因并闭环修复。
4. [ ] 报告与收尾
   - 输出《升级验收报告》与《后续优化待办》。
   - 冻结验收版本号与基线数据。

## 5. 验收标准

- [ ] 关键页面性能达到预设阈值（阈值在本计划执行前冻结）。
- [ ] 关键 E2E 套件稳定通过，失败率满足门禁要求。
- [ ] 升级报告经产品、研发、测试三方评审通过。

## 6. 风险与缓解

- 风险：性能压测环境与生产偏差大。  
  缓解：固定压测环境参数并记录环境元数据。
- 风险：E2E 偶发失败影响结论可信度。  
  缓解：分类标注 flaky 用例并在收口前清零。
