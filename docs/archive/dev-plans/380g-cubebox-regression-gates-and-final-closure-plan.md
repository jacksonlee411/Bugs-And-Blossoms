> 本文不承担功能设计，只负责“什么时候算真正完成”。

## 1. 背景与定位

1. [ ] `380A~380F` 会分别落地数据面、后端、API、文件面、前端与旧资产退役，但没有单独文档承接最终封板条件时，容易留下“局部完成、整体未封板”的状态。
2. [ ] `CubeBox` 目标不是只有新入口可用，而是仓库主链、门禁、测试与文档整体自洽。
3. [ ] 需要一个单独子计划负责最终回归、门禁、证据与完成判定。

## 2. 目标与非目标

### 2.1 核心目标

1. [ ] 跑通 `380` 终态需要的全量门禁、测试与构建验证。
2. [ ] 为旧路径退役、`CubeBox` 正式主链与前向迁移提供最终证据记录。
3. [ ] 定义 `380` 系列从“进行中”切到“已完成”的明确条件。

### 2.2 非目标

1. [ ] 不在本文新增功能。
2. [ ] 不在本文设计 schema、API 或前端 IA。
3. [ ] 不在本文处理单个子计划内部实现细节。

## 3. 关键边界

1. [ ] 封板必须基于实际命令、实际结果与 dev-record 证据，而不是口头声明。
2. [ ] 旧路径退役必须有明确断言，不接受“理论上已删除”。
3. [ ] 文档状态切换必须与测试、门禁、构建结果一致。

1. [ ] 汇总 `380A~380F` 的出口条件与实际落地结果。
